# Houdini 底层架构与技术解析

> 深入剖析 Houdini 的软件架构、运行机制、开发语言与核心操作原理。

---

## 一、开发语言栈

Houdini 采用**多语言分层架构**，不同层级使用不同语言，各司其职：

| 层级 | 语言 | 代码占比 | 用途 |
|------|------|----------|------|
| **核心内核** | **C++** | ~85%+ | 几何引擎、物理求解器、渲染器、内存管理、多线程调度、文件 I/O |
| **HDK 开发套件** | **C++** | — | 与 SideFX 内部构建 Houdini 所用的**同一套 C++ 库**，开发自定义节点、算子、渲染扩展 |
| **VEX** | **VEX**（解释型 DSL） | ~5% | 高性能 SIMD 并行脚本，在几何元素（点/面/体素）上批量执行 |
| **Python（HOM）** | **Python 3** | ~10% | 高层 API：节点创建、UI 脚本、流水线集成、PDG 任务编排 |
| **hscript** | **hscript** | 遗留 | 传统 shell 式脚本，用于表达式和简单命令（逐步被 Python/VEX 取代） |
| **GPU 加速** | **OpenCL / CUDA** | 特定模块 | Pyro、FLIP 等求解器的部分阶段 GPU 加速 |

> [!tip] 关键认知
> **C++ 是 Houdini 的骨架**——整个几何引擎、物理求解器和多线程调度全部由 C++ 实现。VEX 和 Python 是艺术家和 TD 的"操作界面"，底层全部对接 C++。

## 二、核心架构：四层分层设计

```
┌─────────────────────────────────────────────────┐
│           UI 层（Qt + Python 绑定）              │  ← 用户界面
├─────────────────────────────────────────────────┤
│     HOM（Python）/ hscript / VEX                │  ← 用户可编程 API
├─────────────────────────────────────────────────┤
│           算子层（OP_Node 层级树）               │  ← 所有节点的基类
│   SOP / DOP / VOP / COP / CHOP / ROP / LOP / TOP │
├─────────────────────────────────────────────────┤
│       HDK 核心库（GU / GA / UT / VEX 等）        │  ← C++ 构建块
├─────────────────────────────────────────────────┤
│          几何数据模型（GU_Detail）               │  ← 核心数据容器
├─────────────────────────────────────────────────┤
│      操作系统抽象层 + 多线程调度引擎              │  ← 平台适配
└─────────────────────────────────────────────────┘
```

### 2.1 关键 C++ 库与命名空间

| 库/命名空间 | 职责 |
|-------------|------|
| **`GU_Detail`** | 核心几何体容器，管理所有点、面、属性 |
| **`GA_Offset` / `GA_Index`** | 几何属性的偏移/索引类型系统 |
| **`GU_NeighbourList`** | 空间加速结构（空间网格、kd-tree），为 `nearpoint()` 等函数提供底层支持 |
| **`UT_Vector3` / `UT_BoundingBox`** | 数学与工具类型 |
| **`SOP_Node`** | 所有 Surface Operator 节点的基类 |
| **`OP_Node`** | 所有节点类型的终极基类 |
| **`CVEX_Context` / `CVEX_Value`** | VEX 执行的 C++ 运行时桥接 |

## 三、核心运行机制：基于依赖图的"烹饪"（Cooking）模型

这是 Houdini 最根本的运行原理——**基于拉取（Pull-based）的惰性求值依赖图**，内部称为 **"Cooking"**。

### 3.1 脏标记传播（Dirty Propagation）

```
[修改参数] → [标记该节点为 dirty]
                  ↓
    [沿依赖图向下游传播 dirty 标记]
                  ↓
    [受影响的节点全部标记为需要重新计算]
```

- 修改参数时 **不会立即重新计算**整个网络
- 沿**有向无环图（DAG）**向下游传播"脏标记"（dirty flag）
- 所有下游节点被标记为"需要更新"，但**尚未执行任何计算**

### 3.2 按需烹饪（On-Demand Cooking）

```
[视口需要显示结果 / 渲染器需要数据]
            ↓
[请求最终节点的输出数据]
            ↓
[沿依赖图向上游检查：哪些输入是 dirty 的？]
            ↓
[仅对有 dirty 的上游节点执行计算（cook）]
            ↓
[结果被缓存，供后续请求复用]
```

> [!important] 三个核心原则
> - **惰性求值** — 只有被实际请求时才执行计算
> - **最小化重计算** — 只重新 cook 真正变化了的分支
> - **结果缓存** — cook 过的结果被缓存，同一帧内多次请求不重复计算

### 3.3 两种求值模式

| 模式 | 场景 | 优化目标 | 瓶颈 |
|------|------|----------|------|
| **交互模式** | 艺术家在视口中操作 | 低延迟、快速单帧 cook | Amdahl 定律——必须将几乎所有操作并行化 |
| **批量/农场模式** | 最终质量渲染、多帧模拟 | 跨机器并行，吞吐量优先 | 任务调度与数据传输 |

## 四、八大算子上下文（Operator Contexts）

Houdini 将功能按**算子类型**划分为独立上下文，每个上下文有专属节点集：

| 上下文 | 全称 | 职责 |
|--------|------|------|
| **SOP** | Surface Operators | 几何体构建与修改（多边形、体积、NURBS 等）— **最核心** |
| **DOP** | Dynamics Operators | 物理模拟（流体、布料、刚体、粒子） |
| **VOP** | VEX Operators | 可视化编程——节点网络编译为可执行 VEX 代码 |
| **TOP** | Task Operators | PDG 工作流自动化、任务分发、流水线管理 |
| **LOP** | Lighting Operators | USD 场景组装（照明、布局、渲染） |
| **COP** | Composite Operators | 图像处理与合成 |
| **CHOP** | Channel Operators | 程序化动画数据与信号处理 |
| **ROP** | Render Operators | 渲染作业配置与提交 |

> 详见 [[11_操作及名词解释]]、[[04_框架整理]]

## 五、PDG（Procedural Dependency Graph）— 任务编排引擎

Houdini 17.5（2019）引入的架构级升级，由 SideFX 高级软件架构师 **Ken Xu** 主导设计。

### 5.1 核心思想

> PDG 将依赖图的概念泛化——创建高度灵活的**模板图**，节点的多个实例可**同时执行**。这使得任务可并行化、复杂依赖可描述、并能进行更高层次的推理——描述"要发生什么处理"，而非"一个具体的问题实例"。

### 5.2 执行流程

```
┌──────────────┐
│  模板图       │ ← 艺术家描述"要做什么"（抽象工作流）
│  (Template)   │
└──────┬───────┘
       │ 展开（Expand）
       ▼
┌──────────────┐
│  工作项集合   │ ← 自动生成数百/数千个可执行、可监控的工作项
│  (Work Items) │
└──────┬───────┘
       │ 调度分配
       ▼
┌──────────────────────────────────────┐
│  本地核心 / HQueue / Deadline / 云端  │ ← 并行执行
└──────────────────────────────────────┘
```

### 5.3 关键能力

- **Wedging（参数楔入）** — 自动在参数空间上迭代，生成变体矩阵
- **细粒度依赖追踪** — 确保只有受影响的任务被重新执行
- **多机并行** — 本地多核 + 渲染农场 + 云端无缝扩展
- **最小化重计算** — 细粒度依赖追踪确保仅受影响的任务重做

## 六、VEX 与 CVEX：高性能计算桥接

### 6.1 VEX（Vector Expression Language）

VEX 是 Houdini 内置的**解释型、高度多线程的领域特定语言**：

- **SIMD 式执行模型** — 对几何体元素数组进行并行处理
- **不可创建/销毁几何体** — 仅修改现有点/面的属性（保证并行安全）
- **多上下文** — 表面着色器、位移着色器、SOP Wrangle、CVEX 各有专用全局变量

> [!info] 性能秘密
> `nearpoint()` 等空间查询函数底层调用 **C++ 的 `GU_NeighbourList`**，通过 `buildPointGrid()` 和 `buildPointTree()` 构建空间加速结构——这是**纯 VEX 无法复现的性能基础**。

详见 [[06_VEX]]

### 6.2 CVEX：C++ ⇄ VEX 运行时桥接

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  HDK C++     │────▶│  CVEX Context│────▶│  VEX 程序    │
│  程序化代码  │     │  (CVEX_      │     │  (VEX/VOP)   │
│  (开发者)    │     │   Context)   │     │  (艺术家/TD) │
└──────────────┘     └──────────────┘     └──────────────┘
       ▲                                        │
       │           结果沿缓冲区返回               │
       └────────────────────────────────────────┘
```

CVEX 实现了一种优雅的分工：**TD/艺术家用 VEX/VOP 编写算法**，**开发者用 C++ 处理几何体生成和 I/O**——无需重新编译 VEX。

### 6.3 CVEX 核心 API 流程

1. 创建 `CVEX_Context` 对象
2. 通过 `cvex.addInput()` 声明输入参数（vector、float、int、string、matrix 等）
3. `cvex.load()` 加载 VEX 函数
4. `setTypedData()` 将数据缓冲区绑定到输入/输出参数
5. `cvex.run(arraySize, threaded)` — 在所有数组元素上并行执行 VEX
6. 从输出参数缓冲区读取结果

## 七、Houdini Engine（HAPI）— 嵌入式运行时

Houdini Engine 是一个**运行时 API（HAPI）**，将 Houdini Digital Assets（HDA）加载到其他宿主应用中执行：

| 宿主 | 集成方式 |
|------|----------|
| **Unreal Engine** | 原生插件，实时程序化生成 |
| **Unity** | 编辑器内 HDA 加载与 Cook |
| **Maya / 3ds Max** | 桥接插件 |
| **Pixar USD Pipeline** | 自定义 `PxhoudiniEngine` USD Prim Schema + Hydra 生成式程序化 |
| **自定义引擎** | 通过 C API（HAPI）直接调用 |

> Pixar 的集成方案：通过**共享内存传输** VDB 数据，在渲染时进行 Hydra 程序化求值，极大降低工程瓶颈。

## 八、Houdini 与传统 DCC 的本质区别

| 维度 | 传统 3D DCC（Maya/Blender） | Houdini |
|------|---------------------------|---------|
| **编辑模型** | 线性、破坏性历史栈 | 非破坏性、分叉式程序化图 |
| **数据模型** | 单个场景文件，步骤序列 | 节点网络 = 完整构造历史 |
| **并行策略** | 有限的节点内多线程 | 节点内多线程 + PDG 跨机器并行 |
| **可扩展性** | 插件 API | HDK（与核心同源 C++ 库） |
| **修改灵活性** | 回到历史步骤 → 可能破坏后续 | 任意节点改参数 → 自动级联更新 |
| **核心哲学** | "记录做了什么" | "描述如何生成" |

## 九、关键技术组件

### 9.1 VDB（Volumetric Dynamic Grids）

深度集成的 **OpenVDB**（DreamWorks Animation 开发的开源 C++ 库）：

- 使用**分层树结构 + 常量瓦片压缩**高效存储稀疏体积数据
- 核心应用：SDF 实体表面表示、火焰/烟雾/流体模拟、体积布尔运算、多边形-体积互转

### 9.2 HDK 目录结构

HDK 位于 `$HFS/toolkit/`，核心目录：

| 目录 | 内容 |
|------|------|
| `include/` | 所有 C++ 头文件（核心类、几何、VEX 接口） |
| `samples/` | 示例代码（独立应用、CVEX、VEX 扩展、自定义 SOP） |
| `cmake/` & `makefiles/` | 构建系统模板 |
| `slides/` | 深度技术文档（含图解） |

### 9.3 多线程架构

- **交互模式**：利用所有 CPU 核心进行节点内并行（受 Amdahl 定律制约——串行部分决定速度上限）
- **批量模式**：通过 PDG 在多个机器间并行化
- VEX 天然支持 SIMD 风格的并行执行

## 十、总结

Houdini 获得 **3 次奥斯卡科学与技术奖** 的技术根基在于：

1. **统一的程序化范式** — 每个操作都是节点，每个网络都是依赖图
2. **惰性求值 + 脏标记传播** — 高效的最小化重计算策略
3. **PDG** — 将同一套依赖图抽象从单机扩展到数千台机器的农场
4. **VEX/CVEX/HDK 三层可编程性** — 从艺术家到引擎开发者，每个角色都有合适的工具
5. **Houdini Engine** — 让程序化能力嵌入任何宿主应用
