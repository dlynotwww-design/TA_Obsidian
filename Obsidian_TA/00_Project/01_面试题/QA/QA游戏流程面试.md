# 游戏流程面试QA

> 基于 `UE5游戏项目流程` 文档逐节出题。面试官问"你了解游戏开发流程吗"，答的就是这些。

---

## 一、总览与阶段

### Q1: 一个 UE5 游戏项目从立项到上线，分为哪几个大阶段？每个阶段 TA 在做什么？

**答**：

```
立项&预研 ──── TA：技术可行性验证、引擎选型建议、美术风格技术评估
    ↓
全面制作 ──── TA：管线搭建、工具开发、Shader编写、PCG落地、规范制定、日常支持
    ↓
打磨&上线 ─── TA：性能优化、多平台适配、Bug修复、运维工具
```

**展开**：
- **预研期**：验证"这个风格用 UE5 能不能做"——Nanite 支不支持、Lumen 在目标平台跑不跑得动、PCG 框架能不能覆盖需求
- **制作期**：TA 的工作密度最高——HDA 开发、PCG Graph 搭建、母材质库、自动检查工具、规范文档
- **上线期**：Profile→优化→适配→打包。手机上 30fps 稳不稳？PC 上 4K 内存炸不炸？

---

### Q2: "美术风格探索"阶段，TA 怎么参与？

**答**：

不是只让原画自己画。TA 在这个阶段要做：

1. **技术可行性验证**：原画出的风格，引擎能不能实现？——半透明头发在移动端性能如何？大面积水面反射用 Lumen 还是 SSR？
2. **快速原型**：用 Megascans + PCG + 现有材质快速搭一个"风格验证场景"，看效果对不对
3. **性能边界**：确定"电影级体积云"这种需求在目标平台的帧预算上限（如云不能超过 3ms）
4. **工具预研**：如果风格需要特殊的 Shader（如星穹铁道的赛璐珞），提前验证技术方案

---

### Q3: 游戏行业里有句话叫"立项一时爽，优化火葬场"——从流程上怎么避免？

**答**：

**核心思路**：性能约束前置，不要等做完了再优化。

1. **预研期**：定性能预算——场景面数上限、贴图总内存上限、DrawCall 上限、单帧 GPU 时间分配
2. **规范先行**：在美术开始做资产之前就发布面数/LOD/贴图/材质规范（参考模型规范文档）
3. **分阶段检查**：中模→高模→低模→贴图→引擎，每阶段检查，不做完才返工
4. **自动化**：导入时自动跑面数/贴图尺寸/命名检查，不合规直接拦
5. **持续 Profile**：不是最后一个月才跑性能，而是每周跑一次，趋势不对立刻调整

## 二、角色管线

### Q4: 描述一下 UE5 项目的角色美术管线，从概念到引擎上线的完整流程。

**答**：

```
① 概念设计 → ② ZBrush高模 → ③ 低模拓扑+UV → ④ PBR贴图(Substance)
→ ⑤ 绑定+动画 → ⑥ 导入UE5 → ⑦ 材质+物理 → ⑧ 性能验证
```

关键节点：
- **高模阶段**：ZBrush 雕刻百万面，Marvelous Designer 做布料，重点是外轮廓和体块
- **低模+UV**：拓扑降面 → UV 展开 → 烘焙法线/AO/曲率（Marmoset/SP）
- **PBR贴图**：Substance Painter 金属度/粗糙度流程，脸部和皮肤单独处理（SSS）
- **绑定**：骨骼+蒙皮+Control Rig（UE5 引擎内绑定），动捕数据清洗重定向
- **引擎**：材质实例配置 → 物理资产（碰撞/布料/头发） → LOD+Nanite

**参考案例**：心动《伊瑟》50 人团队四阶段并行——概念定位→设计验证(循环3-4次)→并行推进(3D+动画+特效同步)→后期(立绘+配音+PV)，节省 1-2 月/角色。

---

### Q5: 角色管线中，TA 在哪些环节做工具或自动化？

**答**：

| 环节 | TA 工具 | 解决什么 |
|------|---------|---------|
| **建模** | Max/Maya 一键导出脚本 | 统一命名/坐标/单位 |
| **UV** | 自动 UV 检查工具 | UV 利用率/重叠/超出 0-1 |
| **贴图** | SP 导出预设 + 命名规范检查 | 通道正确/格式正确 |
| **绑定** | 自动蒙皮权重工具 | 减少手工刷权重时间 |
| **导入** | 一键导入 + 材质球自动映射 | 手动拖几百个 FBX → 自动批量 |
| **验证** | LOD 自动检查 + 面数报告 | 不合规资产拦截 |

### Q6: MetaHuman 管线比起传统角色管线有什么变化？TA 角色有什么不同？

**答**：

| | 传统管线 | MetaHuman 管线 |
|--|---------|---------------|
| **面部** | 手动雕刻+BlendShape+骨骼 | Mesh to MetaHuman → 自动生成面部绑定 |
| **绑定** | Maya 全手动绑定 | UE5 Control Rig + MetaHuman 标准绑定合并 |
| **迭代** | 改脸要回到 Maya | 引擎内实时调整 |
| **TA 角色** | 搭建绑定工具链 | 定制 MetaHuman 绑定扩展、皱纹贴图增强、LOD 策略 |

## 三、场景/大世界管线

### Q7: 一个开放世界场景从零到上线，管线怎么走？TA 在哪些节点介入最深？

**答**：

```
概念&白盒 → 地形制作 → 程序化场景(PCG) → 手工打磨 → 灯光+后处理 → 性能优化
                ↑                            ↑              ↑
           TA介入最深①                    TA介入最深②    TA介入最深③
```

1. **①地形阶段**：Houdini/Gaea 高度图生成→Landscape 导入→分层材质自动化（Height/Slope/Curvature 驱动）
2. **②PCG 阶段（核心战场）**：HDA 生成器→UE5 PCG Graph→植被散布→建筑生成→道路河流，全部程序化
3. **③优化阶段**：Profile→反推 PCG 参数→LOD 调整→HISM 合并→贴图流送

---

### Q8: 场景管线中"程序化生成"和"手工打磨"怎么分工？边界在哪？

**答**：

| 程序化（PCG） | 手工打磨 |
|--------------|---------|
| 大范围植被散布 | Hero 区域精摆 |
| 建筑群布局 | 视觉焦点建筑 |
| 道路网络 | 关键转弯/交叉口微调 |
| 地形基础形态 | 特殊地貌（剧情场景） |
| 背景山石 | 近景岩石组合 |

**判断标准**：程序化覆盖 80% 的面积但只占 20% 的视觉权重；手工打磨覆盖 20% 的面积但占 80% 的视觉权重——玩家视线停留的地方（任务点/关键景观）必须手工精调。

---

### Q9: "场景拆分为主结构+可复用物件"——这套方法论对 PCG 有什么意义？

**答**：

这是 PCG 设计的前置条件。如果场景资产不模块化，PCG 就做不了：

- **主结构** = 大地形/大建筑框架 → PCG 生成骨架
- **可复用物件** = 窗户/门/栏杆/路灯/花草 → PCG 填充细节

**反例**：如果建筑是"一整个 Mesh 雕刻"，PCG 无法拆解组合 → 只能手工摆放
**正例**：建筑 = 墙体模块 + 窗户模块 + 屋顶模块 + 装饰模块 → PCG 可按规则拼装出无数变体

**TA 的职责**：推动美术团队建立模块化资产库，这是 PCG 管线落地的基建工作。

## 四、特效管线

### Q10: 一个破碎特效的完整管线：从 Houdini 模拟到 UE5 实时播放。

**答**：

```
Houdini 侧：
① Voronoi Fracture → 生成碎片
② RBD 模拟（刚体动力学）→ 碎片碰撞/弹跳
③ VAT 导出 → Position Texture + Normal Texture（EXR格式）
④ 可选：导出 Alembic/USD（非 VAT 方案）

UE5 侧：
① 导入 VAT 贴图 → 设置压缩格式（BC6H/BC7）
② VAT Shader → 读取贴图 → 顶点偏移 → 播放动画
③ 触发逻辑 → 蓝图控制何时播放/停止
④ 性能 → 一个 DrawCall 完成（比骨骼动画便宜几十倍）
```

**VAT vs 骨骼动画的选择**：
- 大量碎片（100+）→ VAT（一个 DrawCall）
- 少量角色级动画 → 骨骼动画（更灵活可混合）

---

### Q11: Niagara 和 Houdini 在特效管线中各自负责什么？

**答**：

| | Houdini | Niagara |
|--|---------|---------|
| **定位** | 离线模拟+预计算 | 实时粒子系统 |
| **擅长** | 流体/烟雾/破碎的物理精确模拟 | 火花/光效/魔法/天气 |
| **输出到 UE** | VAT/Alembic/Flipbook | 原生运行 |
| **迭代** | 改参→重模拟→重导出（慢） | 改参→即时预览（快） |
| **性能** | 预烘焙，运行时便宜 | 实时计算，复杂效果贵 |

**互补原则**：Houdini 做需要物理精度的大效果（龙卷风/海啸/建筑倒塌），Niagara 做需要交互和实时反馈的小效果（技能特效/环境粒子）。

### Q13: "自动化导入+质检报告"——你理想中的自动化管线是什么样的？

**答**：

```
美术提交 Max/FBX/贴图 → 触发自动化流水线：
```

## 六、UE5 技术栈选型

### Q14: Nanite 和 Lumen 都是 UE5 的招牌技术，为什么国内手游厂商很多不用它们？

**答**：

| | Nanite | Lumen |
|--|--------|-------|
| **原理** | 虚拟几何体，百万面直接渲染 | 实时全局光照 |
| **手机问题** | 移动端 GPU 不支持 Virtual Texture 的 Nanite 路径；Android ES 3.1 完全不支持 | 移动端算力不够实时 GI；Lumen 依赖距离场在手机上开销太大 |
| **替代方案** | 传统 LOD + 手动减面 | Lightmap 烘焙 + Light Probe |
| **代价** | 仍需要做 LOD | 需要烘焙时间、需要 UV2 |

**心动《伊瑟》的决策**：立项时 UE5 刚出，移动端没验证 → 用 UE4 稳扎稳打。画质以移动端为基准再向 PC 提升，而非反过来。

**你的简历相关**：你在盘丝洞项目用了 Nanite，面试官可能会问"移动端 Nanite 怎么处理"——答"移动端用 LOD1 替代，PC 端开 Nanite，这是规范和 LOD 策略的一部分"。

### Q15: World Partition 解决了什么问题？传统 Level Streaming 有什么痛点？

**答**：

| | 传统 Level Streaming | World Partition |
|--|---------------------|-----------------|
| **切分** | 手动切关（Level_City_01/02/03） | 自动网格划分 |
| **加载** | 手动触发（蓝图写 Volume 逻辑） | 自动按距离加载/卸载 |
| **协作** | 关卡锁——编辑 Level_City_01 时别人不能改 | **OFPA**：一人一文件，几乎无冲突 |
| **维护** | 策划改了关卡边界 → 所有 Streaming Volume 要重新调 | 自动适应 |

**PCG 场景特别受益**：World Partition + PCG → 只 Cook 加载区域的内容，不用全量生成。

---

### Q16: PCG Framework 和 Houdini 的关系是"竞争"还是"互补"？为什么？

**答**：

**互补**。说"竞争"的人没真正用过大世界管线。

| | Houdini | UE5 PCG Framework |
|--|---------|-------------------|
| **定位** | 通用程序化引擎 | 引擎内实时散布 |
| **计算** | CPU/GPU 离线 | CPU 实时 |
| **深度** | 能做侵蚀模拟/WFC/流体 | 能做散布/过滤/语法 |
| **迭代** | 慢（重 Cook） | 快（实时预览） |
| **产出** | 最终资产 | 场景布局 |

**实际管线**：Houdini 做"造砖"（生成模块资产、复杂算法）→ PCG Framework 做"砌墙"（引擎内散布、组合、调参）。两者不冲突，是上下游关系。

## 七、PCG 专用流程

### Q17: Houdini → UE5 的三条数据通路是什么？各自适用什么场景？

**答**：

| 通路 | 格式 | 适用 |
|------|------|------|
| **Heightmap** | RAW 16-bit | 地形（Landscape Import） |
| **FBX/USD** | 静态 Mesh | 建筑/岩石/道具——Houdini 生成最终资产，引擎当 Static Mesh 用 |
| **Point Cloud** | CSV/JSON/PCG Data | 散布点——Houdini 计算"在哪里放什么"，PCG Graph 执行实例化 |

**选择逻辑**：
- 最终的、不变的资产 → FBX
- 需要引擎内调参的散布 → Point Cloud + PCG
- 地形 → Heightmap + Landscape

### Q18: 混合工作流"四大原则"中，你踩过哪个坑？

**答**（从你的经验中选一个展开）：

**坑例 — "数据流单向"被违反**：
- 做了个 HDA，美术在 UE 里调了参数觉得不满意 → 回 Houdini 改 HDA → 参数变了 → UE 那边之前调的参数全废
- **教训**：HDA 参数一旦暴露给美术就不能随便改，新增参数用版本号管理，旧版本参数保持兼容映射

**坑例 — "属性继承"断裂**：
- Houdini 里设了 `@variant_id` 想做建筑变体，导出 Point Cloud 时忘了带这个属性 → PCG Graph 读到全是默认值
- **教训**：数据导出清单 = 需要哪些属性、每个属性的类型和范围，文档化

### Q19: PCG 团队分工中，你觉得自己最接近哪个角色？如果团队缺了某个角色你怎么补？

**答**：

你的履历显示你最接近 **Houdini TA + 部分程序能力**：能做 HDA、能写 Python、能搭 PCG Graph、能写简单蓝图。

**如果缺程序**：你用蓝图做简单逻辑，复杂底层（如自定义 PCG 节点 C++ 实现）推给程序，但至少能 Clear 地描述需求。
**如果缺地编**：你的建筑学背景能顶一部分——懂空间尺度、会搭白盒、能做材质。
**如果缺策划**：白盒验证阶段你能辅助——快速生成方案让策划跑测。

> 💡 面试时要传达"我知道团队拼图的全貌，我能在自己的拼图之外应急覆盖相邻角色"。

## 八、跨部门协作

### Q20: "策划想要一片森林，美术想要每棵树都不同，程序说面数预算只够 100 棵树"——作为 TA 你怎么解决？

**答**：

这是一个经典的三角矛盾，TA 的职责就是找平衡点。

1. **量化需求**：策划的"一片森林"到底是多大？100m×100m？需要多少棵树？（用白盒快速验证）
2. **美术方案**：不要每棵树都不同——选 5 种基础树 + Seed 变体（随机旋转/缩放）→ 看起来不重复了
3. **程序约束**：100 棵树面数预算 → 每棵 2500 面 → 远树用 LOD1（1250 面）+ LOD2（十字插片 512 贴图）
4. **PCG 方案**：Poisson Disk 均匀散布 → 近处密度高、远处稀疏 → World Partition 只加载玩家周围

**结果**：5 种树 × Seed 变体 = 看起来 50+ 种树 → 实际只用了 100 棵树的面数预算，LOD 后平均每棵 < 1000 面。

---

### Q21: 你写了一堆 PCG 工具，美术不买账怎么办？

**答**：

1. **先问为什么**：是工具难用？生成的不好看？还是不相信程序化？
2. **易用性改造**：参数暴多 → 只暴露 5 个关键参数；要懂 Houdini 才能用 → UE 内一键按钮
3. **用效果说话**：手工摆 500 棵树需要 X 天，PCG 一键生成只需 Y 分钟，且可以随时调整
4. **保留手工出口**：PCG 生成完 → 允许美术手动替换/删除个别物体 → 不剥夺控制权
5. **先小范围试点**：不要求全场景用 PCG，先在一个小区域证明效果，美术自然会主动要求推广

---

### Q22: 程序丢给你一个需求说"把这个算法写进 PCG 节点"，但你觉得有更好的方案，怎么沟通？

**答**：

1. **理解动机**：先问清楚他的目标是什么，不是直接否定方案
2. **分析利弊**：列出他的方案 A 和你的方案 B 的优劣——性能/可维护性/迭代速度/美术友好度
3. **用原型说话**：快速做一个 B 方案的 Demo（哪怕很粗糙），比口头争论有效 10 倍
4. **妥协方案**：如果 B 方案开发成本太高 → 先做 A 方案快速上线 → 迭代期再优化到 B

**关键原则**：TA 不是"对抗"程序，而是帮程序找到更好的实现路径。

## 九、性能优化流程

### Q23: 一个场景帧率不达标，你作为 TA 的排查流程是什么？

**答**：

```
① 定位瓶颈类型
   ├── GPU Bound？→ GPU Visualizer / RenderDoc
   ├── CPU Bound？→ Unreal Insights / Stat Unit
   └── 内存/IO 问题？→ Stat Memory / File I/O Profiler

② 如果是 GPU Bound：
   ├── Shadow Pass 耗时高？→ 减少动态光源/Dynamic Shadow Distance
   ├── Base Pass 耗时高？→ 减少材质复杂度/半透明 Overdraw
   ├── Post Process 耗时高？→ 降低后处理分辨率/简化效果
   └── Nanite 相关？→ 检查半透明/WPO 误用

③ 如果是 CPU Bound：
   ├── DrawCall 太多？→ 合批/HISM/ISM
   ├── Tick 过多？→ 减少不必要的 Actor/Component
   └── PCG Cook 卡？→ 分块生成/异步 Cook

④ 按影响力排序 → 逐个优化 → 每次只改一个变量 → 对比数据
```

### Q24: "Nanite 不支持 WPO（World Position Offset）"——这对植被风动意味着什么？

**答**：

**核心矛盾**：植被必须 WPO 做风动，但 Nanite 不支持 WPO → Nanite 植被不可行。

**解决方案**：
- **方案 A**：植被不用 Nanite，用传统 LOD + WPO → 风动正常但失去 Nanite 优势
- **方案 B**：近景植被传统 LOD+WPO，远景 Nanite（无风动）→ 玩家看远处时风动本来也看不清
- **方案 C**：非植被的 Nanite 物体（石头/建筑）结合植被传统 LOD → 各取所需

**你的简历相关**：盘丝洞项目用了 Nanite → 面试官可能追问"植被怎么处理的？" → 答"植被不用 Nanite，传统 LOD+WPO，石头和建筑开了 Nanite"。

## 十、开放题

### Q25: 如果你加入网易，要从零搭建一套 PCG 管线，你第一周做什么？第一个月做什么？

**答**：

**第一周**：
- 了解现状：现有资产库什么样？美术用什么 DCC？Houdini 用不用？
- 跑通现有场景：手动搭建一个小区域，体验美术现有的痛点和效率瓶颈
- 和程序聊：引擎版本、性能预算、有没有自研限制

**第一个月**：
- 选一个最小可行场景做 PCG 试点（如单一植被散布）
- 搭建 Houdini → UE 数据流（哪怕先手动跑通）
- 产出第一版规范草案：命名/面数/UV/贴图
- 产出第一版自动化检查脚本
- 收集试点反馈 → 调整 → 推广

**关键原则**：不要一上来就搞大而全的框架，用"最小可用→验证→迭代"的方式推进。
