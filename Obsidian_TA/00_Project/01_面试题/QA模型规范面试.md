# 3D场景模型规范 — 面试QA

> 基于 `游戏3D场景模型制作规范` 文档逐条出题，覆盖 TA 面试中与模型规范、资产质量、管线流程相关的高频问题。

---

## 一、流程与管线

### Q1: 描述一个完整的 3D 场景模型从制作到引擎上线的全流程。

**答**：

```
通用物件流程：
中模 → 高模/卡线模 → 低模 → UV + 烘焙 + UV2 → 贴图 → LOD + 碰撞 → 引擎验证 → 提交

建筑流程（多了贴图规划和模块拆分）：
基础白模 → 贴图规划 + 初贴 → 中模(模块拆分) → 高模/卡线模 → 低模 → UV + 烘焙 + UV2 → 贴图 → LOD + 碰撞 → 引擎验证 → 提交

植物流程（灌木/草不需要高模雕刻）：
中模 → 高模 → 低模 → UV + 烘焙 + UV2 → 贴图 → LOD + 碰撞 → 引擎验证 → 提交
（草木灌木类和叶子不需要高模，直接贴图插片）
```

每个阶段结束后需确认才能进入下一步。从烘焙法线开始，必须在引擎中查看效果并截图。

---

### Q2: 中模、高模、低模各自的核心目标和注意事项是什么？

**答**：

| 阶段 | 核心目标 | 关键要点 |
|------|----------|----------|
| **中模** | 确定比例、结构、剪影 | 按人比删减不必要细节；树木可用 SpeedTree/Z球快速造型；要求比例合理、剪影丰富 |
| **高模** | 雕刻大结构、大倒角 | 重点在外轮廓和体块结构；墙体倒角>木头>金属；内凹结构增强切斜角度保证法线烘焙饱满；勿细碎 |
| **低模** | 精简面数、概括形体 | 面数优先外轮廓；细小结构用 Normal Map 表现；底模不倒角（烘焙解决）；剪影优先于细节 |

> ⚠️ 倒角过小会导致烘焙不出法线或法线结构过小，这是新手最常犯的错误。

---

### Q3: 为什么建筑流程比通用流程多了"贴图规划"和"模块拆分"？

**答**：

建筑类通常体量大、贴图需求复杂。如果不提前做贴图规划，容易出现：
- 同一建筑用了多张不必要的贴图 → 增加 DrawCall
- 贴图利用率低 → 浪费内存
- 模块拆分不合理 → 无法复用

**正确做法**：中模阶段同步做贴图规划和粗贴验证美术效果，确认后再拆分模块分别制作，确保一张贴图可复用到多个模块变体上。

---

### Q4: 你作为 TA，如何确保外包/内部美术提交的模型符合规范？

**答**：

从三个层面建立质量保障体系：

**1. 工具层 — 自动化检查**
- Max 内嵌自检插件（如规范文档中的 M5 导出工具），自动扫描命名、面数、材质球、顶点色等问题
- 引擎侧导入时自动校验：LOD 层级、碰撞命名、贴图格式

**2. 流程层 — 分阶段确认**
- 每个阶段（中模→高模→低模→贴图）完成后需截图确认方可进入下一步
- 从烘焙法线阶段开始，必须在引擎内截图提交（规范第 9.2 节明确要求）

**3. 文档层 — 规范可视化**
- 提供面数参考表、命名模板、自检清单
- 截图对比（LOD 效果验证、贴图风格参考）

## 二、软件设置与基础规范

### Q5: Max 中 Reset XForm 是什么？为什么归零后必须执行？

**答**：

Reset XForm 将物体的**变换矩阵烘焙到顶点坐标**，同时重置物体的 Position/Rotation/Scale 为默认值。

**不执行后果**：
- FBX 导出后，引擎中模型缩放/旋转异常
- 法线方向可能错乱（非均匀缩放时尤其严重）
- 碰撞体位置偏移

**规范围明确要求**：坐标归零后 → Reset XForm → 再导出。

---

### Q6: Max 中光滑组（Smoothing Group）和 Maya 软硬边的区别？规范为什么要求以 Max 为准？

**答**：

| | Max 光滑组 | Maya 软硬边 |
|--|-----------|------------|
| **原理** | 给面分配组号(1-32)，同组的面平滑过渡 | 对边标记为软边或硬边 |
| **控制粒度** | 面级别 | 边级别 |
| **烘焙兼容性** | 光滑组直接影响法线贴图烘焙 | 软硬边同样影响，但导出后可能不一致 |

规范要求以 Max 光滑组为准的原因是：
- 场景制作提交以 Max 为准（规范原文）
- 避免 Maya/Max 互导时光滑信息丢失
- **光滑组必须与烘焙法线一致**——不一致会导致法线贴图产生接缝

---

### Q7: UE 和 Unity 的 Normal 贴图有什么区别？烘焙时怎么处理？

**答**：

| | UE（DirectX） | Unity（OpenGL/Maya） |
|--|:---:|:---:|
| **法线 Y 通道方向** | Y-（绿通道向下） | Y+（绿通道向上） |
| **视觉特征** | 凸起处偏浅绿 | 凸起处偏黄绿 |
| **Max 烘焙到引擎** | 不需翻转 Y | **需要翻转 Y 通道** |

**记忆诀窍**：从 Max 烘焙到 Unity → Flip Y；到 UE → 不 Flip。规范原文："场景贴图 Normal 方向对应 Maya 格式（即 OpenGL 格式）"适用于 Unity 项目。

## 三、面数与性能

### Q8: 一棵 8 米高的通用景观树，面数应该控制在多少？为什么？

**答**：

**2500 面以内**（含树干和树叶）。

理由：
- 规范面数表：6-12 米树木控制 2500 面以内
- 手游屏幕上 8 米树木约占画面 15-20%，这个面数在保证剪影效果的同时不会成为性能瓶颈
- 如果同一场景出现 20+ 棵这样的树，总面数约 5 万面，仍在手游可接受范围

**延伸**：实际项目中还需考虑 LOD。6 米以上树木需做 LOD1（减 ≥50%）和 LOD2（截屏插片+512贴图），最终多数视角下的面数远低于 2500。

---

### Q9: 规范中说"大型山石 >12 米"面数反而比"大石头 8-12 米"少（400-1000 vs 800-2000），合理吗？

**答**：

合理。原因：

1. **观看距离不同**：大型山石（>12 米）属于 `structure` 类型，是**中远距离物件**，规范明确"整体制作时注意大方向的形体结构和剪影"，玩家不会近距离观察
2. **归属类别**：大石头 8-12 米归 `architect`（建筑类），可能靠近玩家活动区域；大型山石归 `structure`（大型物体类），是远景
3. **LOD 策略**：大型物体 LOD 间隔 100 米以上，绝大多数时间玩家看到的是 LOD1 或 LOD2，初始面数意义不大
4. **性能分配**：近距离物件面数预算高，远景靠剪影和贴图表现

---

### Q10: 一个 Mesh 为什么不能超过 7000 面？（建筑类规范）

**答**：

1. **手游 GPU 的顶点处理能力有限**：单个 Mesh 过大 → 单个 DrawCall 顶点数过多 → GPU 前端瓶颈
2. **LOD 效率**：超过 7000 面的单个 Mesh，LOD 切换时粒度太粗——要么整体切要么不切
3. **碰撞计算**：碰撞需要包裹独立 Mesh，过大的单 Mesh 碰撞也会过大
4. **规范原文**："单个模型不能超过 7000 面，如超过则根据情况优化或拆分 2 个以上 mesh"——拆分后每个子 mesh 独立材质球，也便于贴图规划

## 四、UV 规范

### Q11: UV1 和 UV2 在引擎中的用途分别是什么？规范对两套 UV 的核心差异是什么？

**答**：

| | UV1（Albedo） | UV2（Lightmap） |
|--|---------------|-----------------|
| **用途** | 颜色/法线/粗糙度等材质贴图 | 引擎烘焙的静态光照信息 |
| **重叠** | **允许重叠** — 对称面可以共用 UV 节省空间 | **不允许任何重叠** — 每个面必须有独立的光照信息 |
| **利用率** | ≥80%，最大化填满 | 尽可能合并，不能浪费 |
| **断开规则** | 硬边处断开 | 接缝必须在朝下/内侧/背面 |
| **间距** | 无硬性像素要求 | 512贴图精度：内部2-4px，边框4-6px |

**核心原则**：UV1 追求**复用和密度**，UV2 追求**无重叠和缝隐藏**。

---

### Q12: UV2 的接缝为什么"禁止断开朝上或视觉直视点处"？

**答**：

Lightmap 在 UV 断开处会产生光照不连续——接缝两侧的像素计算出的光照值可能略有差异。

- **朝上/直视面**：玩家最常看到 → 接缝导致的色差最明显
- **朝下/内侧/背面**：玩家几乎看不到 → 接缝的影响可以忽略

同理，规范要求"填不满时空出右侧或下半区域"——这些区域在 Lightmap 图集中可能被其他模型的 UV 自动填充，而 Unity/UE 的 Lightmap 烘焙会做 UV 间隔保护。

---

### Q13: 什么是"UV 利用率不低于 80%"？如何计算？哪些面可以缩小 UV？

**答**：

UV 利用率 = 实际占用像素面积 ÷ 贴图总面积 × 100%。

**可以缩小 UV 的面**（规范原文）：
- 背面
- 底面
- 看不到的面
- 不重要的小结构

**可以放大 UV 的面**：
- 正面（视觉直视面）
- 贴图细节较多的面
- 模型占比较大的面

**旋转共用注意**：避免过强 Tiling 感——如果同一贴图在同一视角出现太多次重复纹理，视觉上会显得廉价。

## 五、贴图与材质

### Q14: 规范中有两套 Mask 贴图的通道方案，它们的区别是什么？为什么不能混用？

**答**：

| 通道 | 方案 A（MRA — Doc2 项目） | 方案 B（Mask — Doc1 项目） |
|------|--------------------------|---------------------------|
| **R** | **金属度**（0非金属/1金属） | **光滑度**（Smooth） |
| **G** | **粗糙度**（0光滑/1粗糙） | **金属度**（Metallic） |
| **B** | AO | AO |

**RG 通道含义完全相反**。如果混用：
- 方案 A 的贴图放进方案 B 的 Shader → R（金属度）被当作光滑度读取 → 金属物体变得异常光滑，非金属物体反而粗糙错乱
- 反之亦然

这是材质 QA 中最常见的 Bug 之一。TA 需要确保 Shader 的通道映射与贴图通道方案一致。

---

### Q15: 规范中 PBR 材质参数的"安全区间"是什么？为什么要限制？

**答**：

| 参数 | 安全区间 | 禁止值 | 原因 |
|------|---------|--------|------|
| **色彩明暗度** | 0.1-0.9 | 0（纯黑）或 1（纯白） | 纯黑/纯白在 PBR 光照下无法正确表现材质属性 |
| **金属度** | 0（非金属）或 0.8-0.9（金属） | 0.3-0.7 的灰色区间 | 自然界不存在"半金属"；灰色区间在 PBR 中会产生不真实的反射 |
| **粗糙度**（光滑物体） | 0.25 为最低 | 0（镜面） | 除了镜面玻璃外，任何物体都不应完全光滑；0.25 已足够表现抛光金属 |
| **灰色裸金属明度** | ≥0.7 | <0.7 | 金属颜色过暗会丢失金属质感，看起来像深色塑料 |

**规范原文**："切记脏、乱"——各通道颜色上下不超过 20%，避免噪点和脏色混乱。

---

### Q16: 规范中为什么要求单个物体多张贴图时"不能用多维子材质球"？

**答**：

多维子材质球（Multi/Sub-Object Material）在 Max 中虽然方便，但导出到引擎时会出问题：

1. **FBX 导出兼容性**：不同引擎对多维子材质的解析方式不同，可能导致材质 ID 混乱
2. **材质实例管理**：引擎中每个材质球独立管理，多维子材质导入后会拆成多个材质，但关联关系丢失
3. **规范原文**："需要一张贴图一个材质球、并且根据模型 ID（贴图）把模型断开"——断开后每个子 mesh 独立命名、独立材质球，引擎中可以精确控制

**正确做法**：在 Max 中就把不同贴图的部分 Detach 成独立 Mesh，各自分配独立材质球，导出时作为一个 FBX 但包含多个子 Mesh。

---

### Q17: 贴图"法线以大体块为主，减少表面材质颗粒感"——具体怎么理解？

**答**：

这是**风格化 PBR**的核心原则（参考绝区零、星穹铁道）。与纯写实 PBR 的区别：

| | 写实 PBR | 风格化 PBR |
|--|---------|-----------|
| **法线细节** | 砖墙的每一粒沙、木纹的每一根纤维都做法线 | 只做法线大结构（砖缝、木纹方向），不做颗粒级细节 |
| **粗糙度** | 微观变化丰富 | 呈现"彩块化、动漫笔触化"质感 |
| **视觉效果** | 真实但可能"脏、噪" | 干净、颜色丰富、边缘清晰 |

**规范原文**："区别于传统卡通和纯写实风格……去除写实中过于繁杂细碎的细节表现，整体颜色丰富、干净。"

## 六、命名规范

### Q18: 规范中为什么要有两套不同的命名体系？什么场景用哪套？

**答**：

| | 方案一（Doc1/2） | 方案二（Doc3 Unity 项目） |
|--|-----------------|--------------------------|
| **格式** | `Common_物件名_编号` 或 `地图名_build_编号` | `地图名_architect_编号_LOD层级` |
| **复用标识** | 用 `Common_` 前缀标记跨场景复用物件 | 不加前缀，通过地图名前缀区分 |
| **分类标识** | `build`/`tree`/`stone`/`grass`/`terrain`/`small` | `architect`/`bigtree`/`detail`/`structure` |
| **适用引擎** | UE | Unity |

**根本原因**：不同项目、不同引擎的资产管理系统不同，命名规范跟着项目走，不能跨项目混用。TA 的价值在于能快速适配和理解不同项目的命名体系。

---

### Q19: 一个墓地场景有 3 种变体房子共用一张贴图，怎么命名？

**答**：

```
cemetery_build_01a
cemetery_build_01b
cemetery_build_01c
```

- `cemetery` = 地图名
- `build` = 建筑类
- `01` = 编号（共用同一张贴图）
- `a/b/c` = 变体后缀（不同的模型造型）

**贴图命名**：`cemetery_build_01_d` / `_n` / `_m`（三个变体共享同一套贴图）。

## 七、LOD

### Q20: 一棵 8 米的树木，LOD 怎么规划？

**答**：

按照规范：

1. **是否需要 LOD**：8 米 > 6 米，需要
2. **LOD 级数**：面数 ≤ 2500，约 3 级 LOD（LOD0/LOD1/LOD2）
3. **LOD0**：2500 面（完整模型）
4. **LOD1**：≤1250 面（减 ≥50%），保证美术效果与 LOD0 接近
5. **LOD2**：截屏插片制作——引擎中截 LOD1 三视图 → PS 合成 512×512 贴图 → Max 十字插片

**效果验证**：
- LOD0 当前视窗
- 50 米处切换 LOD1
- 再 50 米处切换 LOD2
- 手机端直接使用 LOD1 作为 LOD0（Mobile=1）

---

### Q21: 规范中"LOD0 和 LOD1 的 2U 需保持一致"是什么意思？为什么？

**答**：

LOD0 和 LOD1 共享同一套 Lightmap UV（2U = UV2）。原因：

- LOD 切换时，光照应该连续——如果两套 UV2 不一致，切换瞬间光照会跳变
- 引擎烘焙 Lightmap 时通常只用 LOD0 的 UV2——LOD1 若 UV2 不同，烘焙结果不匹配
- LOD2 不需要 2U 是因为 LOD2 通常是插片/极简模型，距离太远光照细节已不可见

---

### Q22: 手机端模型为什么等于 PC 端 LOD1？

**答**：

规范原文："PC 端和手机端模型同步使用，但是手机端模型是 PC 端的 LOD1 模型"。

原因：
1. **性能差异**：手机 GPU 性能远低于 PC，直接用 PC 的 LOD0 面数太高压
2. **屏幕尺寸**：手机屏幕小，LOD1 在手机上已经足够清晰
3. **画质分档**：通过 Mobile=1 标记，引擎自动选择 LOD1 作为移动端的 LOD0

这意味着 PC LOD 体系需要提前为移动端多留一级——PC 的 LOD1 实际上承担了"移动端 LOD0"的职责。

## 八、碰撞

### Q23: 碰撞为什么必须用简单几何体（Box/凸面），不能用凹面？

**答**：

1. **物理计算效率**：凸面体的碰撞检测（GJK/EPA 算法）是 O(n) 复杂度，凹面体需分解为多个凸面体或使用更复杂的碰撞检测
2. **引擎限制**：UE 的 `UCX_` 前缀碰撞体只支持凸包（Convex Hull）；Unity 中凹面 Mesh Collider 也无法与非凸面 Mesh Collider 碰撞
3. **性能预算**：碰撞每帧都在计算，简单几何体比复杂 Mesh 碰撞快几十倍

**规范原文**："碰撞是由每一个简单几何体制作……只能是 box 或者凸起面，不能是凹面。"

---

### Q24: 花坛/植物类的碰撞为什么要"只有底座有碰撞，植物无碰撞"？

**答**：

这是**游戏玩法设计 + 性能**的综合考量：

| 有碰撞 | 无碰撞 |
|--------|--------|
| 花坛底座 | 植物叶片、枝条 |

- **玩法**：玩家可以靠近花坛但不能穿越底座；枝叶无碰撞避免角色被"卡住"（树叶太密会严重影响行走体验）
- **性能**：植物通常面数高、形状复杂，做碰撞计算量巨大且没有玩法价值
- **规范行为表**：花坛/花盆/植物类 → 只有底座有碰撞

同理，小物件（<家具体积）"不可站立"、栏杆"不可站立"——这些碰撞行为规则由 TA 在设计规范时就预先定义。

### Q25: "简单与复杂"双层碰撞和"将简单碰撞用作复杂碰撞"有什么区别？

**答**：

| | 简单碰撞用作复杂碰撞 | 简单与复杂 |
|--|---------------------|-----------|
| **适用** | 不可交互物件（垃圾桶/栏杆/桌子） | 可交互物件（楼梯/房子） |
| **简单碰撞** | 手动做的 `UCX_` 碰撞体 | 手动做的 `UCX_` 碰撞体 |
| **复杂碰撞** | 复用简单碰撞 | 模型本身作为碰撞（精准碰撞） |
| **计算开销** | 低（只有一层） | 高（需要算两层） |

**例子**：
- 栏杆：不可站立、不可攀爬 → 简单碰撞即可
- 楼梯：玩家需要走上去 → 简单碰撞（粗略）+ 复杂碰撞（精确台阶形状），两层同时生效

### Q26: 如果你发现一批外包提交的模型，法线贴图全部在引擎里看起来反了（凹进去的变凸出来），你会怎么排查和解决？

**答**：

**排查步骤**：

1. **确认引擎**：UE（DirectX 法线）还是 Unity（OpenGL 法线）？
2. **确认烘焙设置**：在 Substance Painter 或 Max 中烘焙时的法线格式是什么？
3. **确认导出设置**：FBX 导出时是否有法线翻转选项被勾选？
4. **定位问题**：如果是 Max 烘焙到 Unity，大概率是**没翻转 Y 通道**

**批量修复**：
- 方案 A（源头上修）：在 SP/Max 中调整导出预设，重新批量导出
- 方案 B（工具批量修）：写 Python 脚本批量翻转 TGA 的 G 通道
- 方案 C（引擎侧修）：在 Shader 中增加 Normal Y Flip 开关（治标不治本，不推荐）

### Q27: 规范中贴图精度写着"1米对应 128-512 像素"，为什么范围这么大？作为 TA 你怎么决定用哪个精度？

**答**：

**范围原因**：不同项目（端游 vs 手游）、不同物件（Hero Asset vs 背景杂物）对贴图精度的需求不同。

| 项目类型 | 推荐精度 | 原因 |
|----------|---------|------|
| 高端手游 | 256-512 px/m | 屏幕小但品质要求高 |
| 中低端手游 | 128-256 px/m | 性能优先 |
| PC/主机 | 512+ px/m | 屏幕大，细节可见 |

**TA 的决策逻辑**：
1. 先定总贴图内存预算（如手游不允许超过 200MB）
2. 按物件重要性和屏幕占比分配精度——Hero Asset 用 512，远景用 128
3. 引擎中 Profile → 根据实际屏幕像素覆盖率反推最佳精度
4. 落地到规范文档中的"贴图精度对照表"

### Q28: 作为 PCG TA，规范文档中的面数参考表对你设计 Houdini HDA 有什么意义？

**答**：

HDA 生成的内容最终要在引擎中运行，面数和 LOD 规范直接决定了 HDA 的生成策略：

1. **Spawn 密度上限**：根据面数表反推每平方米最多放多少物体——例如每棵灌木 200-300 面，场景灌木总面数预算 5 万，则最多生成约 200 棵
2. **自动 LOD 生成**：HDA 中预埋 LOD 生成逻辑，根据物体高度自动判断 LOD 级数（<6 米不做、6 米以上做 2-3 级）
3. **碰撞生成**：HDA 中根据物体类型自动生成碰撞（>0.5m 加碰撞、植物只底座有碰撞）
4. **命名自动生成**：HDA 输出时自动按规范命名（`Common_tree_01_LOD0` / `UCX_tree_01_00`）
5. **性能自动检查**：HDA Cook 后输出面数报告，超阈值自动警告

这就把规范文档从"纸面上"的约束变成了"代码化"的自动执行。

### Q29: 面试官给你看一个场景截图，问"这个场景有什么问题"，你如何用规范文档的框架来分析？

**答**：

按规范文档的六大维度逐一排查：

| 维度 | 检查项 |
|------|--------|
| **模型** | 面数是否合理？剪影是否清晰？有无多余细节？倒角是否合适？ |
| **UV** | 有无贴图拉伸？Tiling 感是否过强？Lightmap 有无漏光/黑斑？ |
| **贴图** | PBR 参数是否在安全区间？有无脏乱噪？金属/非金属区分是否合理？ |
| **LOD** | 切换是否明显？距离设置是否合理？远处是否出现浮空/消失？ |
| **碰撞** | 能否正常行走？有无卡住？栏杆是否可站立？ |
| **风格** | 是否统一？色彩明暗度是否在 0.1-0.9？粗糙度是否呈现亚光？ |

**举例**：如果看到石头反光过强 → 查粗糙度（规范要求最光滑不超过 0.25，亚光为主）→ 如果是金属但看起来像塑料 → 查金属度（非金属=0，不能给灰色中间值）。

### Q30: 规范文档写到什么程度才算"合格"？作为 TA 你如何让美术真的遵守规范？

**答**：

**合格规范的标准**：
- ✅ 有具体的数字阈值（面数/LOD级数/像素间距/PBR参数区间），不是"尽量优化"这种空话
- ✅ 有正反例图片对比
- ✅ 有自动化检查工具，不是纯靠人工肉眼
- ✅ 分阶段交付，每个阶段有 check point

**让美术遵守的策略**：
1. **降低遵守成本**：提供一键检查工具/Max 模板/SP 预设，比违反规范更方便
2. **即时反馈**：提交时自动跑检查 → 有问题当场提示，不要让美术等两天后才发现被退回
3. **正面引导**：展示规范做出来的好效果 vs 不规范的效果对比
4. **教育而非惩罚**：解释每个规范背后的原因（比如"为什么金属度不能给 0.5"）
