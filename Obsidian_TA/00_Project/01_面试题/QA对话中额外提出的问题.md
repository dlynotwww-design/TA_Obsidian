## Q1: Wave Function Collapse 是什么意思？

**一句话**：给你一小块"样本"，WFC 能生成一块任意大的、看起来和样本风格一致的新内容。

**名字来源**：量子力学比喻——粒子从多种可能性的叠加态坍缩为一个确定态。

**算法三步**：

```
Step 1 — 提取规则：从样本学习"什么可以挨着什么"
Step 2 — 迭代坍缩：选熵最小的格子 → 随机坍缩 → 传播约束到邻居
Step 3 — 输出：内容局部满足样本约束，整体布局全新
```

**PCG 应用**：

| 场景 | 样本输入 | 生成输出 |
|------|----------|----------|
| 纹理 | 小块像素拼图 | 无缝大贴图 |
| Tile Map | 小地图块 | 关卡地图 |
| 城市街区 | 建筑排列规则 | 完整城市布局 |
| 室内排布 | 家具布局规则 | 自动摆放房间 |

**优缺点**：

| ✅ | ❌ |
|----|-----|
| 输出天然满足约束 | 计算量大 |
| 不需要训练数据 | 样本必须覆盖所有边界 |
| 可解释、可控制 | 可能陷入无解冲突 |

**面试关联**：网易 JD「AI 结合 PCG」「智能排布室内/城市」——WFC 是可以主动聊的具体方案。

**工具**：Houdini（SideFX Labs WFC Solver）、UE5 社区插件、Unity 开源实现（mxgmn/WaveFunctionCollapse）。

## Q2: Z-Fighting 什么意思？

**定义**：两个面几乎重合时，GPU 无法判断谁前谁后，导致像素交替闪烁的渲染 Bug。

**根因**：Z-Buffer 的非线性精度——越远处的物体深度精度越低，两个面深度值太接近时，浮点精度不够区分。

**表现**：贴花与墙面闪烁、两个 Mesh 拼接处闪烁、LOD 切换时新旧模型重叠一帧。

**解决方案**：

| 方案 | 原理 | 适用 |
|------|------|------|
| Offset 两个面 | 手动错开 0.01-0.1 单位 | 最直接 |
| Depth Bias | Shader 中 `Offset -1, -1` | 贴花/阴影 |
| 提高 Near/Far 比值 | Near 调大、Far 调小 | 场景级 |
| 32-bit Z-Buffer | 精度翻 256 倍 | 引擎设置 |
| Reverse-Z | 反转深度映射 | 现代引擎默认 |
| 不画背面 | 删掉重叠几何体 | 建模阶段 |

**PCG 场景特别关注**：
- 墙面 + 窗户共面 → 给窗户加 0.5cm 外偏移
- 地表贴花 + Landscape 重合 → 加 Z Offset
- 两套 PCG 图表同一位置生成不同物体 → Self Pruning 去重

## Q3: 动态合批与静态合批含义

**静态合批（Static Batching）**：离线合并——标记为 Static 的物体在打包时合并成一个大 Mesh，1 个 DrawCall。代价是额外内存。物体运行时不能移动。

**动态合批（Dynamic Batching）**：运行时合并——每帧在 CPU 上把小 Mesh 顶点合并后提交 GPU。只适用于小 Mesh（顶点 < 300），同材质。每帧有 CPU 开销，物体多了反而负优化。

**对比**：

| | 静态合批 | 动态合批 |
|--|---------|---------|
| 时机 | 离线（打包时） | 运行时（每帧） |
| CPU 开销 | 无 | 每帧合并 |
| 内存开销 | 高（额外存储） | 低（用完丢弃） |
| 物体限制 | 必须 Static | 可移动，小 Mesh |
| 适用 | 不动的大量物体 | 少量移动小物体 |

**PCG 为什么用 HISM 而非合批**：

```
5000 棵树：
静态合批 → 超大 Mesh → 内存炸
动态合批 → 每帧合并 5000 次 → CPU 炸
HISM → 同 Mesh 共用 GPU 数据 → 1 DrawCall，无额外内存，无每帧合并开销
```
