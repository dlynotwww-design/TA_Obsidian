### Q7 🔄 | 半透明渲染闪烁的原因和解决方案？

**复盘来源**：龙游山海 — "光照半透明渲染闪烁" / "半透明问题"

**准备要点**：
- **根因**：半透明物体不写深度 → 渲染顺序依赖排序 → 排序错误导致闪烁
- **方案**：
  - PreZ Pass + EarlyZ → 先画不透明物体写深度
  - 半透明物体按深度从远到近排序
  - 特殊情况（交错半透明）→ 分 RenderQueue 或 切分 Mesh
  - 移动端 → 减少半透明层数、降低 Overdraw

### Q12 📋 | 设计一套程序化地形生成系统，必须覆盖哪些层次？

**JD 映射**：职责 2 — "根据需求实现PCG相关算法，包括不限于：地形，植被，河流，岩壁等"

**必须覆盖的层次**：

```
1. 基础地形 — HeightField
   ├── 大尺度：Perlin/Simplex/Voronoi Noise 控制山脉走向
   ├── 中尺度：侵蚀模拟（水力侵蚀/热力侵蚀）
   ├── 小尺度：细节 Noise 叠加
   └── 分层控制：大→中→小，每层独立参数

2. 地形材质 — 自动分层
   ├── Height → 雪线以上 = 雪（海拔 > N）
   ├── Slope → 陡坡 = 岩壁（坡度 > 60°），缓坡 = 草地（坡度 < 30°）
   ├── Curvature → 山脊 = 岩石，山谷 = 泥土/河流
   └── 输出：Weight Maps（多图层混合）

3. 输出到引擎
   ├── Heightmap (16-bit RAW)
   ├── Weight Maps → Splat Map / Virtual Texture
   └── 可选：Houdini → 直接 HDA Cook 地形 Mesh
```

**加分回答**：讲清楚噪声函数的选择——Perlin 有方向性伪影、Simplex 在各向同性上更好、Voronoi 适合块状地形。

### Q13 📋 | 程序化植被散布的完整方案？从生态分区到具体算法

**JD 映射**：职责 2 — "植被" PCG 算法

**分层散布策略**：

| 层 | 内容 | 算法 | 筛选条件 |
|----|------|------|----------|
| **Layer 1** | 大乔木（森林骨架） | Poisson Disk | 坡度<30°、海拔区间、非水面 |
| **Layer 2** | 灌木（过渡层） | 林缘填充 | 乔木边缘 + 开阔地边缘 |
| **Layer 3** | 花草/地表 | Random + Density Noise | 空地 + 光照充足 |

**算法对比**（必须能讲清楚）：

| 算法 | 适用 | 优点 | 缺点 |
|------|------|------|------|
| Random | 花草 | 简单快速 | 可能扎堆 |
| Poisson Disk | 乔木 | 均匀不重叠 | 计算量大 |
| Density Noise | 过渡 | 自然感强 | 需要调参 |
| WFC | 生态群落 | 满足复杂约束 | 性能开销大 |

### Q14 📋 | 程序化河流路径生成的算法思路？

**JD 映射**：职责 2 — "河流" PCG 算法

```
1. 河流路径生成
   ├── 从高海拔 → 低海拔，沿最陡下降方向（梯度下降）
   ├── 多源头 → 支流汇入 → 干流
   └── 避免局部最小值（Fill Sinks — 填坑）

2. 河流形态
   ├── 宽度 = f(流量) → 下游更宽
   ├── 曲率 = 自然蜿蜒（Sinuosity）
   └── 河床深度随宽度变化

3. 河流影响区
   ├── 侵蚀带（河流两侧 Deform 地形）
   ├── 植被变化（河岸植被带 → 湿度高 → 树更密）
   └── 湿度扩散（影响周围生态分区）

4. Houdini 实现
   ├── HeightField Flow Field → 流向计算
   ├── Find Shortest Path → 河道路径
   ├── PolyFrame + Sweep → 河道几何体
   └── Attribute Transfer → 湿度/侵蚀属性扩散
```

> 📖 详细展开见 [[QA_PCG技术美术面试题集#JD-Q7]]、[[QA_PCG技术美术面试题集#JD-Q8]]（岩壁）

### Q15 📋 | HDA 参数设计中有哪些坑？如何让美术愿意用你的 HDA？

**JD 映射**：职责 3 — "熟练使用Houdini，精通程序化建模"

**五大常见坑**：

| 坑 | 问题 | 解法 |
|----|------|------|
| **参数爆炸** | 暴露 50 个参数 → 美术不敢用 | 只暴露 5-8 个关键参数 |
| **参数耦合** | 改 A 导致 B 失效 | Enable/Disable When 条件控制 |
| **没有默认值** | 空白参数 → Cook 失败 | 所有参数必须有好默认值 |
| **不兼容更新** | 新版 HDA 改了参数 → 旧场景炸 | 版本号 + 兼容映射 |
| **Cook 超时** | 调一个参数等 10 分钟 | 分 LOD Cook + Cache |

**让美术接受的三步**：
1. 先在小范围试点证明效果（如只做植被散布）
2. 保留手工出口（PCG 生成后允许手动替换个别物体）
3. 提供文档 + 视频教程，降低学习成本

### Q21 🔥 | PBR 的核心思想？金属度/粗糙度工作流？

**面经来源**：[[面试经验#四]] — ⭐⭐⭐⭐⭐ "每家必问"

**核心要点**：
- **三个物理原则**：能量守恒 + 微表面模型 + BRDF
- **BRDF 三要素**：
  - **D**（法线分布函数 NDF）—— 微表面朝向分布
  - **F**（菲涅尔）—— 掠射角反射率升高
  - **G**（几何遮蔽）—— 微表面自遮挡
- **金属度安全区间**：0（非金属）或 0.8-0.9（金属），**不能用灰色中间值**（自然界不存在"半金属"）
- **粗糙度最低 0.25**：除了镜面外不应完全光滑
- **色彩明暗度**：0.1-0.9，不能纯黑纯白

**必须熟练到能画图**：金属度/粗糙度的 2×2 矩阵（金属+光滑 / 金属+粗糙 / 非金属+光滑 / 非金属+粗糙）。

### Q23 🔥 | 前向渲染 vs 延迟渲染？区别、优缺点、适用场景？

**面经来源**：莉莉丝、网易互娱等多家考察

| | 前向渲染 (Forward) | 延迟渲染 (Deferred) |
|--|-------------------|---------------------|
| **光照计算** | 每个物体 × 每个光源 | 先渲染 G-Buffer，再统一计算光照 |
| **光源数量** | 多光源性能差（O(n×m)） | 大量动态光源效果好 |
| **半透明** | 天然支持 | 需要额外 Pass（Forward Pass） |
| **MSAA** | 天然支持 | 不直接支持（G-Buffer 是屏幕空间） |
| **显存** | 低 | 高（G-Buffer 多张 RT） |
| **适用** | 移动端/VR/少量光源 | PC/主机/大量动态光源 |

**Forward+**（中间方案）：分 Tile 做光源剔除，兼顾 MSAA 和多光源。适合移动端。

**G-Buffer 压缩**：BC6H/BC7、降精度（FP16→R8G8B8A8）、Octahedron 编码法线。

### Q24 🔥 | 实时阴影技术栈 —— Shadow Mapping 原理？PCF/PCSS/VSSM 区别？

**面经来源**：[[面试经验#四]] — ⭐⭐⭐⭐ "高频"

**Shadow Mapping 原理**：
1. 从光源视角渲染深度图（Shadow Map）
2. 主渲染时，将每个像素变换到光源空间
3. 比较深度：当前深度 > Shadow Map 深度 → 在阴影中

**反走样与软阴影**：

| 技术 | 原理 | 特点 |
|------|------|------|
| **PCF** | 对 Shadow Map 周边多个采样取平均 | 简单、但有锯齿 |
| **PCSS** | 基于遮挡物距离动态调整采样范围 | 近处硬、远处软 |
| **VSSM** | 用方差近似深度分布（切比雪夫不等式） | 快但可能漏光 |

**阴影问题**：阴影痤疮（Shadow Acne）→ Depth Bias 解决；彼得潘现象（Peter Panning）→ Bias 太大导致。

### Q25 🔥 | 移动端 GPU 抗锯齿方案？MSAA/FXAA/TAA 区别？

**面经来源**：腾讯 IEG — "移动端 GPU 抗锯齿方案"；字节跳动

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **MSAA** | 每个像素多次采样（子像素） | 效果好、几何边缘 | 贵、不支持延迟渲染 |
| **FXAA** | 后处理 — 检测边缘 → 模糊 | 极快、1ms 内 | 整体画面略模糊 |
| **TAA** | 多帧混合 + 抖动采样 | 效果好、处理 Shader 锯齿 | 鬼影（Ghosting） |
| **SMAA** | FXAA 升级版（更智能的边缘检测） | 效果比 FXAA 好 | 比 FXAA 略贵 |

**移动端推荐**：TAA（UE5 默认）+ 适当锐化。MSAA 在移动端开销太大，FXAA 太糊。

### Q26 🔥📋 | DrawCall 优化？HISM/ISM/Static Mesh Component 区别？

**面经来源**：[[面试经验#四]] — 合批技术高频
**JD 映射**：[[岗位JD]] 要求 2 — 熟悉 UE 引擎

| 类型 | DrawCall | 适用场景 |
|------|:---:|------|
| Static Mesh Component | 1 个/物体 | 少量独特物体（<10） |
| ISM | 同 Mesh 合并为 1 | 中等量重复物体（10-1000） |
| HISM | ISM + 层级剔除（自动 LOD+Culling） | 大量植被/石头（万级以上） |

**为什么 PCG 用 HISM 而非静态合批**：

- HISM → 共用 GPU 数据 → 1 个 DrawCall，无额外内存，自动层级剔除

### Q27 🔥 | C++ 基础 —— 虚函数机制？智能指针？右值引用？

**面经来源**：[[面试经验#四]] — 渲染向 TA 必考

**虚函数机制**：
- 通过**虚函数表（vtable）**实现多态——每个类一张 vtable，对象存 vtable 指针
- 构造函数**不能**是虚函数（构造时 vtable 尚未建立）
- 析构函数**应该**是虚函数（确保派生类正确析构）

**智能指针**：
- `shared_ptr`：引用计数，线程安全（计数增减是原子操作），但指向对象不线程安全
- `unique_ptr`：独占所有权，零开销
- `weak_ptr`：打破 `shared_ptr` 循环引用

**右值引用与移动语义**：
- `std::move` 不移动任何东西，只做类型转换（左值→右值引用）
- 移动构造函数接管资源而非拷贝——适合大容器（如 `std::vector` 扩容）
