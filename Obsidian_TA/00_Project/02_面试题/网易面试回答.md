## 一、针对岗位

### Q1: 如何设计一个生态群落？

**核心思路**：从宏观到微观的分层设计，用数据驱动取代手工摆放。

**第一层 — 地形与气候驱动**：
```
输入数据：
├── HeightField（海拔）
├── Slope（坡度）
├── Aspect（坡向 → 日照）
├── Moisture（湿度 → 模拟水流扩散）
└── Temperature（温度 → 基于海拔梯度递减）

→ 这些数据共同定义"生态位"（Ecological Niche）
```

**第二层 — 生态区划分**：
- 用上述数据的组合将地形分类：
  - 高海拔+陡坡 → 岩壁生态（苔藓+碎石+矮松）
  - 中海拔+缓坡+向阳 → 阔叶林生态
  - 低海拔+近水 → 河岸生态（柳树+芦苇+湿润植被）
  - 山脊+迎风面 → 灌木过渡带
- 分类用 `Attribute Remap` + `Attribute Promote` 将连续值映射为离散生态区标签

**第三层 — 群落分层散布**：
```
Layer 1: 大乔木（森林骨架）
├── 算法: Poisson Disk Sampling（保证最小间距）
├── 密度受 Moisture + Temperature 调制
└── 物种选择基于生态区标签

Layer 2: 灌木/小乔木（过渡层）
├── 算法: 基于大乔木位置的边缘填充
├── 聚集度用 Clustered Noise 控制
└── 避开大乔木的 Canopy Range

Layer 3: 地表植被（填充层）
├── 算法: 纯随机 + Density Noise
├── 排除乔木/灌木/岩石的 Bounds
└── 花草种类按生态区随机变体

Layer 4: 特殊元素
├── 枯木/倒木（基于 Slope + 随机概率）
├── 岩壁藤蔓（陡坡 + 暴露岩面）
└── 蘑菇/菌类（高湿度 + 低光照区域）
```

**第四层 — 竞争与演替**（进阶）：
- 用 WFC 或约束求解模拟树种竞争：橡树和松树不能在同一微生态位共存
- 演替阶段参数：幼林（密度高+高度矮）→ 成熟林（密度低+高度高）
- 可通过全局 Seed 控制不同演替阶段

**Houdini 实现要点**：
```vex
// 示例：基于生态区标签的物种选择
string biome = point(0, "biome", @ptnum);
float moisture = point(0, "moisture", @ptnum);
float slope = point(0, "slope", @ptnum);

if (biome == "riverbank") {
    // 河岸：柳树为主
    @species = moisture > 0.7 ? "willow" : "alder";
} else if (biome == "forest") {
    // 森林：按海拔分树种
    @species = slope < 25 ? "oak" : "pine";
}
```

**输出到 UE**：
- HDA 输出 PCG Point Cloud → UE PCG Framework 直接读取
- 每个点携带：`Transform`、`Species`、`Scale`、`Seed`
- 引擎侧用 Static Mesh Spawner 按 `Species` 属性分发不同 Mesh

### Q2: 如何设计一个城市系统？

**核心思路**：分层级生成——"路网定骨架、地块定分区、建筑定细节、装饰定氛围"。

**第一层 — 路网生成**：
```
1. 主干道（Arterial Roads）
   ├── 输入：城市边界 + 关键节点（城门/广场/码头）
   ├── 算法：L-System 或 基于 Tensor Field 的路径生成
   └── 结果：定义城市大分区

2. 次干道（Collector Roads）
   ├── 在主干道围合区域内细分
   ├── 棋盘式（格网城市）或 放射式（欧洲老城）
   └── 参数控制：路网密度、曲线度

3. 支路（Local Roads）
   ├── 填充剩余空间
   └── 确保每个地块可达
```

**第二层 — 地块划分**：
```
道路网络 → 面 → Parcel 划分：
├── 沿主干道：大型商业/公共建筑地块
├── 沿次干道：中型住宅地块
├── 内陆：小型住宅地块
└── 特殊地块：广场/公园（手动标记或规则识别）
```

**以长安城为例**（结合你的长安项目经验）：
- 长安城是**里坊制**——规整的棋盘式路网
- 每个"坊"是一个封闭的街区单元（500m×500m 左右）
- 坊内再细分巷道和建筑组团
- **这是建筑学知识直接转化 PCG 的典型案例**——你理解真实城市的空间逻辑

**第三层 — 建筑生成**：
```
每个地块 → 生成建筑：
├── 地块 → Building Footprint（退线/缩进）
├── 建筑类型匹配（商业/住宅/工业/公共）
├── 立面生成：
│   ├── Shape Grammar（底层商铺+上层住宅+屋顶）
│   ├── 或 模块化拼装（预设墙体/窗户/屋顶模块）
│   └── Grammer 语法示例：[Wall, Window]* → [ShopFront, [Wall, Window]*] + Roof
└── 变体控制：Seed → 随机风格/高度/颜色
```

**第四层 — 街道家具与植被**：
```
沿道路自动布置：
├── 路灯（间隔 30m，偏移 1.5m）
├── 行道树（间隔 8m，树种按街区风格）
├── 消防栓、长椅、垃圾桶
└── 交叉口特殊处理（交通灯/路牌/转角建筑退让）
```

**技术实现路径**：

| 方案 | Houdini 为主 | UE PCG 为主 | 混合 |
|------|:-----------:|:----------:|:---:|
| 路网 | ✅ 算法强 | 手动 Spline | Houdini 生成路网 → UE |
| 地块 | ✅ 几何处理 | 弱 | Houdini → PCG Data |
| 建筑 | ✅ 程序化建模 | Shape Grammar | Houdini 生成变体 |
| 散布 | 点云导出 | ✅ PCG Spawner | 点云传入 UE |
| 迭代 | Cook 慢 | 实时预览 | Houdini 预生成 → UE 微调 |

**推荐混合方案**：Houdini 做路网+地块+建筑变体生成 → 导出点云+Level Instance → UE PCG 做最终散布和实时调整。

**对比你的实际项目**：
- 长安城 2km²：Houdini 生成路网 + 里坊结构 → UE 中散布建筑模块
- 600m×600m 概念城市：手动制作街区单元模块 + PCG Level Instance 散布
- 如果放到商业游戏中，核心改进是：**增加手工干预入口**（美术能在 UE 中局部修改而不破坏全局规则）

### Q3: 模型流程规范

**问题的本质**：面试官想知道你是否理解"标准化"——不只是能做出来，而是能让团队复用的规范。

**回答框架**：从命名 → 结构 → 材质 → UV → 碰撞 → LOD → 导出全链路。

```
一、命名规范
├── 文件命名：{项目}_{类型}_{描述}_{变体}_{LOD}.fbx
│   例：LY_BDG_House_Chinese_A_LOD0.fbx
├── Mesh 命名（引擎内）：
│   SM_{类别}_{描述}_{变体}
│   例：SM_BDG_Wall_Stone_01
├── 材质命名：
│   M_{主材质}_{变体}
│   MI_{主材质}_{场景}_{变体} (Material Instance)
└── 纹理命名：
    T_{Mesh名}_{通道}_{类型}
    例：T_Wall_Stone_D (Diffuse), T_Wall_Stone_N (Normal), T_Wall_Stone_ORM (Occlusion+Roughness+Metallic)

二、模型结构规范
├── Pivot 点规则：
│   ├── 建筑：底面中心
│   ├── 窗户/门：几何中心（方便 Shape Grammar 替换）
│   ├── 路灯/柱子：底部中心
│   └── 植被：底部中心（根部）
├── 层级结构（便于程序化替换）：
│   Root (SM_Building_01)
│   ├── Walls (SM_Wall_01, SM_Wall_02...)
│   ├── Windows (SM_Window_01...)
│   ├── Doors (SM_Door_01...)
│   └── Roofs (SM_Roof_Gable_01...)
└── 模块化尺寸标准：
    墙体模块 = 4m × 3m（高）
    窗户 = 1m/2m 宽 × 1.5m 高
    → 确保任何组合都能无缝拼接

三、材质与UV规范
├── UV Channel 0：主贴图（0-1 空间，允许超出，用 UDIM 管理）
├── UV Channel 1：Lightmap（必须在 0-1，无重叠，引擎自动生成或手动）
├── 材质 ID：按物理材质分配（玻璃=1，木材=2，石材=3…）
├── Texel Density：统一 512px/m 或 1024px/m（项目级别统一）
└── PBR 验证：BaseColor 50-240 sRGB，Roughness 检查

四、碰撞体规范
├── 简单碰撞：Box/Sphere/Capsule（引擎自动生成或自定义 UCX）
├── 复杂碰撞：Use Complex as Simple（仅特殊需要，性能昂贵）
├── 碰撞命名：UCX_{MeshName}_01
└── 程序化生成时的碰撞策略：
    建筑 → Box 碰撞 / 植被 → 无碰撞（仅 Foliage）/ 路灯 → Capsule

五、LOD 规范
├── LOD0：100% 面数，0-15m
├── LOD1：50% 面数，15-40m（Simplify 或 Imposter 开始）
├── LOD2：25% 面数，40-80m
├── LOD3：Imposter/Billboard，80m+
└── PCG 资产额外要求：
    每种变体都要有完整 LOD 链（Houdini 可自动生成 LOD HDA）

六、导出规范（Houdini → UE）
├── FBX：静态 Mesh（含 LOD 组）
├── JSON/CSV：Point Cloud Transform 数据
├── Heightmap：16-bit RAW（Landscape）
├── HDA：需要实时 Cook 的复杂资产
└── USD：多软件协作的大型管线
```

**面试时的"闪光句"**：
- "我做规范不是为了约束美术，而是让他们少踩坑——规范是踩坑踩出来的经验文档化"
- "我在盘丝洞项目中整理了内部技术知识文档，把常见的渲染异常排查流程写成了 Check List"

## 二、Houdini 基础知识

### Q4: PDG 是什么？怎么用？

**PDG（Procedural Dependency Graph）** 是 Houdini 的任务调度和自动化系统。

**核心概念**：
```
传统 Houdini：手动调参 → Cook → 导出（一次一个）
PDG：定义"要做什么"（Work Item）→ PDG 自动批量执行
```

**架构**：
```
TOP 网络（Task Operator）
├── Work Item：最小工作单元（如"生成第 5 栋建筑"）
├── Scheduler：调度器（本地/农场/云）
├── Mapper：将参数数组展开为 Work Items
└── Partitioner：将结果合并

常用节点：
├── Wedge：参数扫描（"分别用 seed 1-100 生成 100 栋建筑"）
├── ROP Fetch：调用 ROP 输出节点
├── ImageMagick：图像后处理
├── Python Processor：自定义 Python 任务
└── HDA Processor：批量调用 HDA
```

**PCG 中的实际应用**：
```python
# 场景：需要生成 500 栋建筑的变体 + 渲染缩略图

# PDG 流程：
Wedge(seed=1-500)
  → HDA Processor(Cook 建筑 HDA，传入 seed)
  → ROP Fetch(导出 FBX)
  → ROP Fetch(渲染缩略图)
  → ImageMagick(缩略图裁剪/加标签)

# 结果：一键运行，自动出 500 个 FBX + 500 张缩略图
```

**与 PCG 管线的关系**：
- PDG = 管线自动化的"引擎"
- 可以在夜间自动 Cook 整个场景的所有 HDA
- 结合 CI/CD：提交 HDA 更新 → 自动触发 PDG Cook → 自动导入引擎 → 自动截图对比

### Q5: Point 与 Vertex 的区别？各层级的作用？

**这是 Houdini 最核心的概念区分问题**。

```
层级关系（从低到高）：
Vertex（顶点）  ⊂  Point（点）  ⊂  Primitive（面）  ⊂  Detail（全局）
```

**Point vs Vertex 的关键区别**：

| | Point | Vertex |
|--|-------|--------|
| **含义** | 空间中的一个位置点 | 面上的一个引用点 |
| **唯一性** | 全局唯一，每个点只有一个 | 一个 Point 可以被多个面引用 |
| **数量关系** | 通常 ≤ Vertex 数量 | ≥ Point 数量 |
| **存储** | 空间位置 + 通用属性 | UV + 面相关属性 |
| **共享** | 多个面共享同一点（焊接点） | 不共享，每个面有自己的 Vertex |
| **Wrangle** | Point Wrangle 操作 | Vertex Wrangle 操作 |

**具体例子**（一个立方体）：
```
立方体 = 8 个 Point（8 个角）
       = 24 个 Vertex（6 个面 × 4 个顶点）
       = 6 个 Primitive（6 个面）
       = 1 个 Detail

为什么是 24 个 Vertex 而不是 8 个？
→ 因为每个面的 UV 不同！同一点在相邻面上需要不同的 UV 坐标。
   Point 存储位置，Vertex 存储 UV——所以必须分开。
```

**各个层级的实际作用**：

```
Detail（全局属性）
├── 用途：存储全局参数（Seed、全局缩放、版本号）
├── 访问：detail(0, "seed")
├── 例子：整个城市工具的城市名称、坐标原点
└── Wrangle: Detail Wrangle

Primitive（面/体）
├── 用途：存储面的属性（材质 ID、光滑组、法线模式）
├── 访问：prim(0, "material_id", @primnum)
├── 例子：一栋建筑的每个墙面 → 不同材质 ID
└── Wrangle: Primitive Wrangle

Point（点）
├── 用途：存储空间位置 + 通用属性（颜色、密度、法线）
├── 访问：point(0, "P", @ptnum)
├── 例子：植被散布点 → @density、@species、@pscale
│         这是 PCG 最常用的层级！
└── Wrangle: Point Wrangle

Vertex（顶点）
├── 用途：存储 UV、顶点颜色、顶点 Alpha
├── 访问：vertex(0, "uv", @vtxnum)
├── 例子：面的 UV 展开
└── Wrangle: Vertex Wrangle
```

> **面试加分点**：能讲清楚"为什么需要 Vertex"——因为同一点在不同面上可能有不同的 UV/法线。这是建模基础但很多人面试时说不清楚。

### Q6: VEX 的使用经验

**VEX（Vector Expression Language）** 是 Houdini 的高性能脚本语言，类似 C 语言语法。

**在 PCG 中的核心能力**：

```vex
// 1. 条件筛选 — PCG 最常用
if (@height > 200 && @slope < 30) {
    @density = 1.0;
} else {
    @density = 0.0;
}

// 2. 属性计算 — 驱动程序化规则
float dist = distance(@P, chv("center"));
@density = 1.0 - smooth(0, chf("radius"), dist);  // 中心密→边缘疏

// 3. 随机化 — 变体控制
float seed = @ptnum * 123.456;
@pscale = fit01(rand(seed), 0.8, 1.2);           // 随机缩放
@orient = quaternion(rand(seed+1) * 2*PI, {0,1,0}); // 随机旋转

// 4. 空间查询 — 避让逻辑
int near_pts[] = pcfind(0, "P", @P, 2.0, 5);  // 找 2 米内的点
if (len(near_pts) > 0) {
    @density = 0.0;  // 太近了，不放置
}

// 5. 噪声驱动 — 自然感
float noise = noise(@P * chf("noise_scale") + chf("seed"));
@density = fit01(noise, 0.3, 1.0);  // 噪声调制密度
```

**VEX vs Python 在 Houdini 中的选择**：

| 场景 | 用 VEX | 用 Python |
|------|:------:|:--------:|
| Wrangle 逐点操作 | ✅ | ❌ 太慢 |
| 创建/连接节点 | ❌ | ✅ |
| 批量文件处理 | ❌ | ✅ |
| 噪音/数学计算 | ✅ | ✅ |
| HDA 参数自动化 | ❌ | ✅ |

**我的使用体会**：
> VEX 是 PCG TA 的"手写笔"——你能用 VEX 写出来的逻辑，就是你能精确控制的逻辑。Wrangle 节点是 Houdini 和拖节点之间最本质的区别。

### Q7: Houdini 各个模块

```
Houdini 核心模块：

SOP (Surface Operators) — 几何体操作
├── 最常用模块，PCG 核心所在
├── 功能：建模、修改、生成几何体
├── 代表节点：Attribute Wrangle, Copy to Points, PolyExtrude
└── PCG 用途：程序化建筑、地形处理、植被散布

DOP (Dynamic Operators) — 动力学模拟
├── 功能：刚体、柔体、流体、粒子、布料
├── 代表节点：RBD Solver, Pyro Solver, FLIP Solver
└── PCG 用途：废墟破碎、藤蔓物理缠绕、水流侵蚀模拟

VOP (Vex Operators) — 节点化 VEX 编程
├── 功能：可视化编程方式写 VEX/Shader
├── 类似材质编辑器
└── PCG 用途：复杂噪声叠加、自定义材质

ROP (Render Operators) — 输出
├── 功能：渲染、导出、缓存
├── 代表节点：File Cache, FBX Export, Mantra
└── PCG 用途：批量导出 FBX/USD、烘焙贴图

COP (Compositing Operators) — 图像处理
├── 功能：合成、图像处理
└── PCG 用途：PSD 解析、Mask 生成、贴图处理

TOP (Task Operators) — PDG 任务调度
├── 功能：自动化、批量、管线
└── PCG 用途：批量 Cook 建筑变体、自动化管线

CHOP (Channel Operators) — 动画通道
├── 功能：音频、运动数据处理
└── PCG 用途：程序化动画驱动

LOB (LOP — Lighting Operators) — USD 场景管理
├── 功能：USD 原生编辑、场景组装
└── PCG 用途：多 DCC 场景数据交换
```

**PCG TA 的核心模块比**：
```
SOP:  ██████████ 90% — 工作主战场
VEX:  ████████░ 80% — 嵌入在 SOP 中
TOP:  ████░░░░░ 40% — 管线自动化
COP:  ███░░░░░░ 30% — PSD/贴图处理
DOP:  ██░░░░░░░ 20% — 物理模拟辅助
```

### Q8: Houdini 使用了多长时间？

**参考回答（根据你实际情况调整）**：

> "我从建筑学的参数化设计起步，最早接触的是 Grasshopper（3年+），这让我建立了**数据流和程序化思维**。2022年左右开始系统学习 Houdini，到现在大约 **3-4 年**。
>
> 前半年主要在学基础——SOP 节点、VEX 语法、属性传递的核心逻辑。后面逐步深入到程序化建筑、地形、植被散布等实际 PCG 场景。目前我能独立从零搭建一套城市/建筑/植被的 Houdini → UE 管线。

### Q15: 教程知识点 — WFC

> 面试官在问你对 WFC 的理解深度。

**参考回答**：

> "WFC（Wave Function Collapse）是我自学的一个重点算法。我从 2D 的纹理合成理解起，然后扩展到 3D 城市布局。
>
> **核心原理**：
> 1. 给一小块样本 → 提取'什么东西可以挨着什么东西'的约束规则
> 2. 初始化：每个格子是所有可能状态（原型）的叠加
> 3. 迭代：选熵最小的格子（可能性最少→最容易确定）→ 随机坍缩为一个状态 → 传播约束到邻居（删掉不兼容的状态）
> 4. 重复直到全部确定 或 出现冲突
>
> **我在 3D 城市布局中的实验**：
> - 样本：一个手工设计的 3×3 建筑街区
> - 每个格子记录：建筑类型（住宅/商业/办公）+ 高度 + 朝向
> - WFC 扩展到 10×10 街区 → 自动生成保持局部风格一致、全局布局随机的大型城市
> - 输出到 Houdini + 可视化
>
> **WFC 的局限性**（理解这一点面试很加分）：
> - 计算量大（N×N 网格 × M 种可能状态）
> - 样本必须覆盖所有边界条件（否则无解崩溃）
> - 全局结构弱（WFC 只保证局部一致，不了解'城市该有中心轴线'这种全局约束）
> - → 所以实际项目中通常结合**规则系统**（L-System/Shape Grammar）做宏观结构，WFC 填充微观细节
>
> **工具**：我主要用 Houdini SideFX Labs 的 WFC Solver 做实验，也在 UE 中用过社区 WFC 插件。"

### Q17: 道路工具 — 需求、细节、难点、交叉口处理

**需求**：为 UE 项目开发一个道路生成工具，能快速生成带细节的道路系统。

**具体功能**：
```
输入：Spline 路径
输出：完整道路（路面 + 标线 + 路肩 + 行道树 + 路灯 + 消防栓）

功能列表：
├── 多宽度支持（主干道 12m / 次干道 8m / 支路 5m）
├── 宽度渐变（如转弯处加宽）
├── 自动标线（虚线/实线按道路等级）
├── 行道树自动间隔散布
├── 路灯自动排布（偏移 1.5m，间隔 30m）
├── 消防栓/垃圾桶等街道家具
└── 交叉口自动处理
```

**技术细节**：
```
1. Spline → 路面 Mesh
   ├── PolyFrame：沿 Spline 计算切向/法线
   ├── Sweep：沿路径挤出道路截面
   ├── UV 自动生成：路面 UV 沿路径展开（方便标线贴图）
   └── 宽度渐变：用 Spline 的 Width Attribute 驱动截面缩放

2. 标线：
   ├── 虚线：UV 空间 Mod 运算 → 周期 Mask
   ├── 实线：道路边缘偏移 0.2m 生成
   └── 材质混合：用 Vertex Color 标记标线区域

3. 行道树/路灯：
   ├── Resample Spline（固定间距）→ 采样点
   ├── 偏移：沿法线方向偏移 2m（避开路面）
   ├── 高度检查：采样 Landscape 高度
   └── 随机微调：±0.5m 间距抖动（避免过于整齐）
```

**最大难点 — 交叉口**：

> "交叉口是道路工具最难的部分。核心挑战是：**两个 Spline 的 Mesh 拼到一起怎么才能无缝？**
>
> **我的方案（分三步）**：
>
> **Step 1 — 检测交叉口**：
> - 用 `Intersection Analysis` 检测两条 Spline 的交叉点
> - 在交叉点前后各预留一段"过渡区"（如 5m）
>
> **Step 2 — 生成交叉口模型**：
> - 预制作交叉口模块：十字路口 / 丁字路口 / 转盘 三种预制 Mesh
> - 根据交叉角度自动选择 + 旋转匹配
> - 路口 Mesh 的 UV 从中心放射状展开（和标准段不同）
>
> **Step 3 — 解决接缝**：
> - 标准段路面在距交叉口 0.5m 处做 `Clip` 裁切
> - 交叉口模块边缘做 0.5m 的过渡区（UV 渐变混合）
> - 材质处理：交叉口和标准段用**同一材质实例**，通过 World Position 做过渡混合，减少接缝视觉差异
> - 顶点焊接：交叉口边缘顶点 Snap 到标准段边缘顶点（Houdini Fuse SOP）
>
> **接缝材质处理的具体做法**：
> - 路面材质使用 World Aligned Texture（三平面投影），而非 UV 贴图
> - 在交叉口和标准段的过渡区，用 Distance Blend 混合两种 UV 模式
> - 效果：肉眼几乎看不出接缝"

### Q18: 道路工具怎么优化？

**性能优化**：

| 优化项 | 方案 | 效果 |
|--------|------|------|
| **Mesh 合并** | 同材质路段合并为一个 ISM Component | DrawCall 从 N 条路 → 1 |
| **行道树 HISM** | 所有同树种树木合并 HISM | 数千棵树 = 几个 DrawCall |
| **LOD** | 路面 3 级 LOD（全细节→简化→极简），路灯 2 级 | 远处性能大幅提升 |
| **Cull Distance** | 路灯/消防栓 80m 外不渲染 | 减少远处 DrawCall |
| **Spline 采样密度** | 根据曲率自适应采样（直路段疏，弯道路段密） | 减少无用顶点 |
| **Imposter 远景** | 50m+ 的行道树用 Billboard Imposter | 远景观感不变，面数降 90% |

**功能优化**：
```
- 交互式编辑：在 UE 中拖动 Spline 控制点 → 道路实时更新（Houdini Engine 实时 Cook）
- 局部重生成：只 Cook 被修改的 Spline 段，不改动区域不动
- 回退机制：每次修改保存 Seed + 参数快照，方便对比和回退
- 美术微调入口：生成后允许美术手动替换个别路灯/树，下次 Cook 时保留手动修改
```

### Q20: 城市工具放到游戏中，有什么改进？

> 面试官想看你对"学术项目 → 商业产品"的思维转换。

**参考回答**：

> "如果把我之前做的城市 PCG 工具放到商业游戏中，我会做以下改进：
>
> **1. 从'全自动生成'到'可控生成'**
> - 之前：输入参数 → 全自动 → 输出整个城市
> - 改进后：生成全局框架 → **美术可以在引擎中手动修改任何局部** → 重新 Cook 时保留手动修改
> - 技术上需要：属性标记（Auto/Manual）+ 局部 Lock 机制
>
> **2. 性能预算前置**
> - 之前：不太考虑性能，只关注视觉效果
> - 改进后：每个街区/建筑都有性能标签（面数预算、材质复杂度、碰撞复杂度）
> - 生成时自动控制：近景全细节 → 中景简化 → 远景 Imposter
>
> **3. LOD 全链路**
> - 之前：没有 LOD
> - 改进后：Houdini 自动为每种建筑变体生成 3-4 级 LOD
> - 引擎侧按距离自动切换
>
> **4. 全局变体 + 局部特色**
> - 之前：整个城市一种风格
> - 改进后：不同街区可以有不同风格（Seed + Style Tag），但要保证过渡区自然
> - 比如：老城区→新城区 建筑密度和高度逐渐变化
>
> **5. 运行时 Player 感知适配**
> - 之前：做方案效果图，固定视角
> - 改进后：考虑玩家移动速度 → 根据速度调整 Streaming 范围
> - 快速移动时加载范围更大但 LOD 更低
>
> **6. 结合游戏玩法**
> - PCG 不只是视觉，还要考虑 gameplay
> - 比如：自动标记可攀爬建筑（生成 NavMesh 信息）、自动放 Cover Point（射击游戏掩体）
>
> **总结一句话**：学术项目的 PCG 追求'生成得好看'，商业游戏的 PCG 追求'性能可控 + 美术可控 + 玩法适配'。"

### Q24: Claude Code 基本原理，用了什么模型？

**参考回答**：

> "Claude Code 是 Anthropic 推出的 AI 编程助手，核心原理：
>
> **模型层**：
> - 基于 Anthropic 的 Claude 系列模型（主要是 Claude Sonnet 和 Claude Opus）
> - 底层是 Transformer 架构的大语言模型

>
> **工作原理**：
> 1. **上下文理解**：读取整个项目文件结构 + 相关代码文件
> 2. **工具调用**：不只是聊天，可以实际执行命令（读文件/编辑文件/运行测试/git 操作）
> 3. **Agent 模式**：可以自主规划多步操作，像人类程序员一样"读代码→理解→修改→验证"
>
> **与我作为 TA 的关系**：
> - 我用它来处理重复性的代码工作（如写批量处理脚本、模板代码）
> - 但它不知道 Houdini 的特殊节点逻辑、不知道 UE 管线的上下文——这些必须我来把关
> - 本质上它是我的一层'外脑'，提升效率但不能替代判断"
