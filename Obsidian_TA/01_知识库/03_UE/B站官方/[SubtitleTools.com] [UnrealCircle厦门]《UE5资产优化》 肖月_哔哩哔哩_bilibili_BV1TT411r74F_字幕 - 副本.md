# UE5资产优化 — 肖月 @ UnrealCircle 厦门

## 一、模型优化

模型优化从**场景整体规划**到**单个资产优化**，分为三个层面：
1. **剔除（Culling）** — 只提交场景内实际可见的面进行渲染
2. **Draw Call 控制** — 控制发送到 GPU 的批次数量
3. **面数/LOD 控制** — 由远及近按屏幕占比控制三角面密度，减少 Overdraw

---

### 1.1 剔除（Culling）

#### 非 Nanite 物体的剔除

逐物体剔除，按执行顺序：

| 阶段 | 说明 |
|------|------|
| **距离剔除** | 按预设可见距离剔除 |
| **视锥剔除** | 判断物体是否在当前相机视锥内 |
| **遮挡剔除** | 计算物体是否被前后遮挡 |

- 剔除参考对象：物体的 **Bounding Box**
- 调试命令：
  - `r.VisualizeIncludedPrimitives` — 查看被遮挡剔除的物体（绿色 Bounding Box）
  - `stat InitViews` — 定量性能检查
    - 上半部分（`View Visibility Inclusion Culling`）：剔除计算的耗时 → **剔除效率**
    - 下半部分：剔除前后 Primitive 数量变化 → **剔除效果**

**制作建议：**
- 检查不同物体的 Bounding Box 设置是否合理
- 大场景按视觉重点设置不同物体的**最远绘制距离**（Actor 细节面板）
- 远处细小道具、地平线植被 → 用距离剔除提前优化掉

#### Nanite 物体的剔除

- 剔除单位从 Component 细化到 **Cluster**（相邻三角面组成）
- 大部分在 **GPU** 上高效完成
- 基于**屏幕占比**动态调整三角形密度和 Cluster 大小
- 调试：Nanite Cluster 视图模式

---

### 1.2 Draw Call 控制

#### Draw Call 的拆分规则

**非 Nanite 物体：**
- 不同 Actor / Component → 不同 Draw Call
- 同一资产内：不同 LOD 层级 / 不同材质 → 不同 Draw Call
- ISM（Instanced Static Mesh）：按 Instance 种类全场景合批，效率高很多
- HISM（植被等）：还会考虑 Instance 的 LOD 层级

**Nanite 物体：**
- 渲染管线对相同材质的模型更高效合批
- Draw Call 压力显著减小
- 拆分模型时更多从**内存占用**和**加载效率**考量

#### 模型拆分策略

| 策略 | 优点 | 缺点 |
|------|------|------|
| **整体 Mesh** | Draw Call 少 | 相机外/内部遮挡区域无法有效剔除 |
| **拆得过细** | 剔除效果好 | Draw Call 数量太多 |
| **模块化 + 复用** | 引擎 Auto Instancing 动态合批 | 适合重复的窗户、墙面等 |

**建议工作流：**
1. 从**易于编辑的模块化模型**做起，不做提前优化
2. 场景物件/Draw Call 过多时，再考虑 Actor 合并
3. 优先合并：距离近 + 面数低 + 材质相同的模型
4. 工具：**Merge Actors**（引擎内置）

> **关键思路：多考虑 Instance 方案**
> 
> 以《黑客帝国》项目为例：
> - 程序化生成，场景中有百万级 Instance
> - 实际工程中模型数量并不多
> - 可拼出各种风格的建筑，Draw Call 负担轻
> - 柱子和护栏 → 相同模组 Instance 拼接（节省内存和包体）
> - 桥面（有弧度无法模组拼接）→ 按 World Partition Grid Cell 切割，Streaming 加载

#### 单资产层面：控制材质 ID 数

- 不同材质 = 不同 Draw Call（无论是否 Nanite）
- 重点：控制模型材质槽数
- 远处低精度 LOD 上做材质简化和合并

### 1.3 Overdraw 与 LOD

#### Overdraw 产生原因

三角形密度超过屏幕像素精度 → 一个像素包含多个三角形 → 光栅化浪费

**GPU 影响：**
- Base Pass（G-Buffer 生成）
- PrePass（深度渲染）
- 阴影贴图绘制

调试命令：`stat GPU`，视图模式：**Quad Overdraw**（不透明硬表面最高显示到 4 级/绿色）

#### LOD — 最直接的优化方案

**传统 LOD：**
- 引擎自带自动减面工具（按百分比生成多级 LOD）
- 但自动减面不适用于所有模型（如建筑 → 无法维持轮廓框架，需手动低模）
- 不同物件视觉重要性不同 → LOD 级数和减面程度不同
  - 级数太高 → Draw Call 和内存受影响
  - 级数太低 → Overdraw 优化效果差 / 切换时视觉跳变明显

**项目实践：**
- 按部件类型在 Config 文件中设置一系列 LOD 组，规范级数和减面规则
- 便于批量调整和管理资产

#### Nanite 的"无极 LOD"

- 不再以物体级别做 LOD 切换/整体减面
- 按不同区域的屏幕占比，**逐 Cluster 做面数降级**
- 屏幕三角形密度分布更合理
- 省掉大量美术工作，无视觉跳变
- 调试：Nanite 三角面模式

---

### 1.4 Nanite 启用与注意事项

#### 启用建议

- PC 端：**能用尽用**
- 可批量处理
- 引擎有 Nanite 工具可细致筛选（面数过滤、不支持的材质类型过滤）

#### 效率不高的场景

1. **拓扑不均匀的模型** — 尽量均匀，但也要看具体外观，不强求
2. **多层紧密堆叠的 Mesh** ⚠️
   - 低视角/远距离时，Nanite 无法判断哪个面在最上层 → 大量 Overdraw
   - 还会导致光追加速结构遍历缓慢
   - 调试：Nanite Overdraw 视图模式
   - 制作建议：尽量避免穿插，批量检查 → 布尔合并操作

#### 数据量优化

1. **减少不必要的硬边/光滑组（裂开的边）：**
   - 硬边 → 多一份法线存储 → 增加顶点数
   - 裂开的边无法组成 Cluster → 影响剔除和减面效率
   - 顶点数尽量**小于**面数（顶点数 = 面数 × 3 说明所有面都裂开了）

2. **UV 控制：**
   - UV 块切太碎 → 数据量大增
   - 建议：高模制作中**提前展 UV**（模型大型出来后就开始），后期加细后做 UV 传递
   - 扫描资产 → 先减面 → 展 UV → 传递到高模

3. **不要使用过多 UV Set**，非必要不勾选"生成 Lightmap UV"

## 二、材质优化

材质消耗主要在 **GPU** 和**内存**。

调试：材质复杂度视图模式（红色区域应尽量避免）

---

### 2.1 主要性能瓶颈

1. **大量半透明材质**
   - 粒子贴片在相机前多层叠加
   - 即使材质计算简单，半透明需要多 Pass 渲染 → 消耗迭代增长

2. **材质指令数过高**
   - 调试：材质编辑器 **Stat 窗口**
   - 指令数来源：节点计算 + 光照模型计算
   - 不接受光照（Unlit）最简单
   - 复杂光照模型（毛发、眼睛、次表面皮肤）指令数更高

**PC 端指令数参考标准：**（非绝对，以项目视觉重点为准）

---

### 2.2 优化建议

1. **规划母材质，不要用单一大材质覆盖所有效果**
   - 量产前规划几种母材质
   - 用 Static Switch 节点做材质分支

2. **注意节点开销差异：**
   - 相对节省：加减乘、Clamp
   - 相对昂贵：除法、指数运算、三角函数、**Noise 节点**
   - Noise → 用烘焙 Noise 贴图替代

3. **远处/低光泽度物体：** 勾选材质 **Fully Rough** 选项

4. **半透明材质着重简化：**
   - 粒子贴片受光照影响不明显
   - 尽量把效果烘焙到贴图 → 运行时用 Unlit 渲染

---

### 2.3 UE 5.1 Nanite 材质更新

| 版本 | Nanite 支持 |
|------|------------|
| 5.0 | 仅静态硬表面，固定不可编程光栅化路径 |
| 5.1 | 可编程光栅化路径 → 支持 WPO（顶点动画）、Mask 材质 |

但相对于普通硬表面，这部分材质仍有相当大的性能消耗：

**WPO 优化：**
- 开启 Optimized WPO 模式
- 仅在必要时激活顶点动画（远处植被风动注意设置开启距离）
- 材质内顶点位置计算逻辑尽量简化
- 复杂多级风动 → 烘焙到 VAT（Vertex Animation Texture），简化为单次贴图采样

**Mask 材质优化：**
- 开销比 WPO 还大
- 透明区域的 Overdraw 无法避免
- 使用应更加节省
- 《堡垒之夜》案例：植被树叶 → 评估后选用**高面数模型**代替 Mask 材质 + 面片
  - 详见下方技术文章

## 三、光影优化

静态烘培光照运行时开销极小，略过。聚焦**动态光**。

---

### 3.1 直接光优化

#### 光照计算

1. **光源类型：** 面积光更昂贵，应尽量节省
2. **灯光剔除：** 局部光（Point Light、Spot Light）按场景大小设置最远绘制距离
3. **控制灯光影响范围，减少重叠区域**
   - Point Light → 衰减半径
   - Spot Light → 锥角大小
   - 调试：灯光复杂度视图模式（红色重叠 = 计算更复杂）

#### 阴影优化

调试：关注 `Stat GPU` 中 **Shadow Depth** 的消耗。

核心思路：限制绘制范围 + 绘制精度 + 绘制物体个数

**阴影开销排序：**

| 光源 | 开销 | 原因 |
|------|------|------|
| Point Light | 最大 | 需 6 个面（前后左右上下）投影 |
| Directional Light | 中等 | 单面投影但影响范围大 |
| Spot Light | 最节省 | 单面投影 + 锥角范围限制 |

**优化建议：**
- 严格限制开启阴影的 Point Light 数量
- 粒子特效中 Instanced 的 Point Light → 一般不需开启阴影
- Directional Light 注意远处物体是否要开启投影
- 视情况用 Spot Light 代替 Point Light 做局部照明
- 非 Nanite 物体：用命令快速限制需要投影的物体范围

---

### 3.2 Virtual Shadow Maps (VSM)

> UE5 PC 端推荐组合：Nanite + VSM（效果和效率最佳）

#### 为什么需要 VSM

传统阴影贴图（基于深度投影）的问题：
- 灯光视角绘制阴影贴图 vs 屏幕空间采样 → **视角偏差**
- 导致屏幕不同区域阴影精度不一致
- 精度不足 → 锯齿、自遮挡等 artifacts

**VSM 方案：**
- 把阴影贴图切割成 **Page**，根据物体到相机距离做精度控制
- 尽量统一屏幕像素精度和阴影贴图精度
- 只追踪足够的 Mip 精度进行采样，避免浪费
- 效果 vs CSM：全场景阴影细节明显提升

#### VSM 配置

**查看模式：**
- Page 视图 — 观察每个 Page 的覆盖区域和精度
- Clipmap 视图 — 颜色标记精度等级
- Mask 视图 — 实际绘制阴影区域

**精度设置：**
- 局部灯光：默认 16K，8 级 Mip
- Directional Light：Clipmap 存储，默认 17 级，每级 16K
- `r.Shadow.Virtual.MaxPhysicalPage` — 调整纹理池大小
- 分别有 Directional Light / Local Light 的 LOD 偏移和精度降级设置

#### VSM 缓存机制 — 核心性能优势

- 静态物体 → 缓存 Page（视图模式下大片的**绿色**区域），不需每帧重绘
- 需要关注 → **红色**区域（缓存更新区域，性能消耗峰值来源）
- 以 Page 为单位更新
- 更新触发：光线/视角快速变化、动态物体覆盖
- 策略：静态物体和动态物体**分开缓存**（蓝色 = 仅更新了动态缓存）

⚠️ 注意动态物体的 **Actor Bounds Scale** — 引擎以 Bounds 覆盖区域确定更新哪些 Page，设置过大会造成巨大性能压力。

#### VSM 软阴影

- 采用 SMRT 光线追踪方式
- 设置较大光源开角 → 接触点硬阴影到远处软阴影自然过渡
- 优化点：
  - 追踪光线数量控制
  - 每根光线采样数控制
  - 噪点不明显时适当降低光线数
  - 采样数越高 → 半影更柔和，但性能代价更高

### 3.3 Lumen 间接光

Lumen 提供真实动态 GI 和反射，取代之前低质量的屏幕空间方案。

> 核心：Lumen 的间接光照贡献都来自 **Lumen Scene**

---

#### 3.3.1 光追精度

**软件追踪（Distance Field）：**
- 调整 Mesh Distance Field 精度（模型资产编辑器中）
- 薄物体（墙面、屋檐）可能漏光 → 用厚模型做 DF 替代（仅用于光追，不渲染）

**硬件追踪：**
- 追踪真正的三角面
- Nanite 使用面数较低的 **Fallback Mesh**
- 面数低、有棱角的模型 → 注意 Fallback Mesh 不要减面太狠

---

#### 3.3.2 Lumen 中的材质

- 物体表面光照信息靠自动放置的 **Mesh Card** 捕捉
- 命令：`Card Placements` 查看
- Surface Cache 模式查看捕捉效果（黄色/紫色 = 未被渲染到 Lumen Scene）

**提高间接光贡献：**
1. 提高后处理盒子 **Lumen Scene Detail**
2. 针对复杂结构模型 → 增加 **Mesh Card** 数量（需看情况，卡太多影响效率）
3. 凹面多/带空洞的复杂模型 → 离线拆分模型

---

#### 3.3.3 硬件光追效率优化

1. **控制动态物体面数：**
   - 光追依赖场景内加速结构（BVH）
   - 角色动画、破碎粒子每帧重建 → 注意面数
   - 静态模型顶点动画同理 → 大多数情况不需微动级对间接光照产生影响
   - 命令：设置顶点动画的光追开启距离

2. **限制视窗外物体追踪范围：**
   - Lumen 需要屏幕外物体贡献间接光，但可用命令限制追踪范围
   - 《黑客帝国》案例：小模块搭建的建筑整体设为光追组 → 整体剔除 → 提高效率

3. **简化 Lumen 中材质计算：**
   - 使用 **Ray Tracing Quality Switch** 节点拆分：
     - 上方入口 → 屏幕空间 G-Buffer 材质计算
     - 下方入口 → Lumen 光追到达物体表面时的材质计算
   - Lumen 侧材质大幅简化：
     - 省掉细节法线
     - Roughness < 0.4 → 光追效率极低
     - 直接设 Roughness = 1 提高效率

---

## 总结

| 模块 | 核心优化方向 | 关键工具/技术 |
|------|-------------|--------------|
| 模型 | Culling → Draw Call 控制 → Overdraw/LOD | Nanite（推荐 PC 端尽用）、Merge Actors、ISM/HISM |
| 材质 | 指令数控制 + 半透明简化 + 母材质规划 | Static Switch、Fully Rough、VAT、Unlit |
| 直接光 | 影响范围控制 + 阴影范围/精度/物体数控制 | VSM（推荐配合 Nanite） |
| 间接光 | Distance Field / Fallback Mesh 精度 + Mesh Card + 光追简化 | Lumen、Ray Tracing Quality Switch |

> **核心思想：** UE5 引入的 Nanite + VSM + Lumen 三大技术体系在 PC 端已形成最佳组合，它们相互配合，能在效果和效率上达到很好的平衡。但仍有不适用的场景（模型堆叠、Mask 材质等），需要美术层面针对性优化。
