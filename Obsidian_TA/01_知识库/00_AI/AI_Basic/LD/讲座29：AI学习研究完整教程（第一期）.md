## 一、模型选择：普通人如何选模型

### 核心观点

对于大部分日常编程和问答工作，85分模型和95分模型**没有显著差异**。普通人无需追求最贵最强的模型。

### 推荐方案

**国内用户性价比之选**（80-90分模型，满足大部分需求）：

- 智谱
- Moonshot（月之暗面）
- 阿里
- DeepSeek（直接充值即可）
- 小米、快手、MiniMax、极夜星辰

> 以上厂商的 Coding Plan / 订阅会员均可购买使用，性价比高，能满足普通人的日常 coding 需求。

**顶级模型的定位**（95分）：

- 功能确实强大，但对普通人来说**过于昂贵**
- 日常 agentic coding 任务中，85分和95分差异不明显

### 类比

拿世界级数学难题给陈省身、丘成桐、陶哲轩 → 他们能解决，普通人不行。
但初中数学题 → 不需要天才，普通人也能解决。

大部分人的需求（写前端、写全栈 APP）属于"初中数学"级别，不需要最聪明的模型。

## 二、AI Agent 学习路线

### 第一课：手写 Minimum SWE Agent

每个计算机系 / AI 专业大学生的**必修课**，就像过去手写编译器、操作系统、数据库一样。

**三步走**：

1. **看懂**：参考 Claude Code、Codex 等开源/闭源实现，理解它们的代码
2. **想明白**：自己设计 agent 的功能模块
3. **实现出来**：手写代码并让它跑通测试

**最小版本的核心功能**：

| 功能 | 说明 |
|------|------|
| Task Scheduling | 将 prompt 拆分为多个 task |
| Prompt 拼接 | 正确构造给大模型的 input |
| Function Calling | 正确喂入 function calling / MCP |
| Output Parser | 正确解析大模型 API 的 output |
| 命令执行 | 在 terminal 执行 bash 命令、HTTP request 等 |
| 文件读写 | 基本的 IO 操作 |

> 最小的 SV agent 只需要 **terminal 执行 + 文件读写** 就能正确运行。

### 进阶完善：从"马车"到"高铁"

写完最小版本后，需要对照当代最先进的开源实现（Claude Code、Codex、Open Code、Kimi Code 等），学习以下**进阶机制**：

1. **Long Term Memory** — 如何实现长期记忆，包括 sub-agent 的记忆管理
2. **Multi-Agent 调度** — 如何在 TUI 中管理多个 sub-agent，读取状态、监控输出
3. **Background Tasks 管理** — 管理后台运行的多个 terminal 任务
4. **Skills / Plugins** — 如何让 agent 正确读取 skill 的 description/title，在适当时机选取正确的 skill
5. **Context Auto Compression** — 上下文过长时的手动/自动压缩
6. **Loop & Workflow 控制** — 判断 loop 何时该停，不能无限 loop 或随意终止
7. **Sandbox 环境控制** — Docker 中启动、监控、停止运行环境
8. **MCP 自由配置**
9. **可视化与可观测性** — 哪些给用户看、哪些自己判断、哪些不能给用户看
10. **TUI 设计** — 设计更合理的人机交互体验

> 先把这些学好 → 尝试实现 → 形成自己的理解 → 才能提出创新。

## 三、非程序员的简易 Agent 方案

### 适用人群

- 不在大厂、不做核心 agent 业务
- 对 AI 没有深度信仰
- 不在 AI startup
- 不会写代码 / 写不利索

### 方案：Claude Code Headless + Temp Folder

**步骤**：

1. 给 Claude Code 绑好银行卡
2. 建立一个临时文件夹（temp folder）
3. 把所有相关资源放进去：
   - 文档、表格、报表、图表、报告
   - SQLite 数据库、Access 数据库、PDF 扫描件
   - Markdown、代码、图片
   - 指令 prompt、configure、skills
4. 写一个 **Node.js / Bun.js / Python** 脚本
5. 从 terminal 调用 `claude code headless` 模式
6. 讲明指令，文件夹默认开启权限（`--dangerously-skip-permissions`）
7. 配置好 authentication 和 MCP
8. **成功/失败** 都将状态和输出存储到 JSON file
9. 等 process 结束 → 读取 JSON → 下一步

> 这个方法对 99% 的非程序员来说，是最简单、最强大、最可靠的**宇宙级通用 AI agent** 方案。

### 例外情况

如果你：
- 懂上下文管理
- 有自己领域的 LLM 省钱技巧
- 完全懂你的业务场景
- 能设计出比 Claude Code + 手工 function calling 更好的方案

→ 那你可以自己设计 agent，否则用上面的通用方案。

## 四、AI Agent 的能力边界

### 核心原则

> **人不能解决的问题，agent 也一定解决不了。**

### Agent 不是神

Agent 不能做的事：

- 解决哥德巴赫猜想（人类也解决不了）
- 预测明天彩票中奖号码
- 预测微软股票涨 5% 还是跌 10%
- 证明黎曼猜想
- 预测今年最赚钱 / 跌最多的股票
- 预测 3 年后墨西哥 GDP

> 这些违反**因果论**和**信息论**的事情，agent 即使给出答案也不可信——它只是汇总了三五个网页而已。

### 判断 Agent 能否解决一个任务

必须同时满足：

1. **人有没有能力完成** — 有合理的流程和方法论能系统性地解决
2. **有足够的上下文和数据** — 文档、图表、数据库等齐全
3. **有正确科学的流程** — agent 完全参照流程和方法执行

> 没有上下文 = 无法完成；没有正确流程 = 风险极大。

## 五、AI Agent 的完美应用场景

### 四大条件（Goal-Driven 前提）

| 条件 | 说明 |
|------|------|
| 1. 封闭环境 | 不与外界有过多的接触和交流 |
| 2. 完整运行/编译/模拟/仿真 | 本地可以完整执行 |
| 3. 无限试错 | 搜索空间巨大 |
| 4. 可精确验证 | 输出可被客观验证 |

### 五大完美场景

#### 1. 编程

- 写算法、数据结构、编译器、数据库算法
- 本地运行 → 本地测试 → 本地报错 → 迭代
- 或在稳定的云环境（AWS 节点）中运行
- **封闭环境 + 可验证 = agent 舒适区**

#### 2. 数学（Lean 4）

- Lean 4 构建证明 → 本地编译 → 本地检查
- 可参考 arXiv 论文、其他人的 paper
- 难度比编程高很多，但模型迭代后会提升
- 代表项目：Exosome 等（已融资 1 亿美元以上）

#### 3. 电子设计 / EDA / 芯片设计

- Verilog、VHDL、System Verilog 等 HDL
- 本地设计 → 本地 synthesis → 本地 RTL 仿真 → 本地 verification
- 过去三四十年 EDA 工具已非常成熟
- 仿真之后仍需留片和现场测试，但整体是完整闭环

#### 4. MATLAB + Simulink

- 每个 Toolbox 代表一个领域：机械、热力学、流体、量子计算等
- 100-200 个 Toolbox × agent = 每个领域都可能诞生新的自动化研究工具
- 只要买了注册号，就能本地进行完整仿真
- **MATLAB 是留给 AI agent 最好的礼物**

#### 5. CAD / 机械设计

- AutoCAD + Lisp 脚本操控
- 也包括 G-code（数控机床）、PLC（控制系统）、大型工业系统
- 完整的设计 → 模拟 → 仿真环境

### 不适合的场景

- **金融/股票市场**：公开场景、有对手、无法精确测试验证

## 六、Multi-Agent 分析

### 大前提

- Token 一定会越来越便宜
- Multi-agent 并发度越来越高
- Token 速度越来越快
- LLM 模型越来越聪明

> Multi-agent 是你能相信的**唯一 post-training / agent scaling 方式**。用 1000 倍成本实现 10-100 倍提升，是极佳投资。

### ❌ 不适合 Multi-Agent 的场景

| 场景 | 原因 |
|------|------|
| **编程 / Coding** | 多人贡献代码 = code management 灾难，七手八脚一团糟；更适合单人一条道走到黑 |
| **角色串联流水线**（如 MetaGPT） | 产品经理 → 架构师 → 程序员 → Code Review → QA → 再回程序员，**串行效率低**，不如单 agent 持续工作 |
| **公司架构模拟**（如 Crew AI、Work Body） | 七嘴八舌互相说话，没有产出，谁来决定谁来管理？过去两三年公认的**超级大天坑** |

### ✅ 适合 Multi-Agent 的场景

| 场景 | 说明 |
|------|------|
| **大规模并行任务** | Map → 多个 agent 并行处理 → Reduce 汇总，架构干净 |
| **具体应用** | 大规模网页搜索、爬虫、文献调研、survey、数据清洗、并行客服、大规模审核 |

> 关键：任务足够并行、足够独立拆分、汇总架构清晰。

## 八、总结

1. **选模型**：国内普通模型性价比最高，日常够用
2. **学 agent**：手写 minimum SWE agent 是大学生第一课
3. **进阶**：对照先进实现学习 memory、multi-agent、TUI 等机制
4. **非程序员**：用 Claude Code headless + temp folder 即可
5. **能力边界**：人解决不了的，agent 也解决不了
6. **完美场景**：编程、数学(Lean4)、EDA、MATLAB、CAD
7. **Multi-agent**：适合并行任务，不适合串行和公司模拟
