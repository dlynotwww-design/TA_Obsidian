## 一，内容库构成

Content Library 内容库，为 [Houdini城市生成器](https://zhida.zhihu.com/search?content_id=250765938&content_type=Article&match_order=1&q=Houdini%E5%9F%8E%E5%B8%82%E7%94%9F%E6%88%90%E5%99%A8&zhida_source=entity) 提供数据。

### 1.1，美术资产制作

**在开始美术资产的批量制作前要制定好引擎的标准光照环境以便进行着色的校准** ！：

![](https://pic3.zhimg.com/v2-7f21baf5ea712b9b6e7739ee7499edda_1440w.jpg)

“标准光照环境”应该是国内外各个游戏项目的一个共识了，严谨的灯光着色验证可以减少后期很多麻烦。而在美术设计资产时也遵循明确的“方法论”，在CitySample中设计围绕三个问题考虑：

1. 它是由什么材料做成的？【即建筑是花岗岩的，还是砖石的，城市造景在建筑用料上不是随便决定的，一切都有理可循，材料与建筑样式和城区规划决定了一座城市的风格】
2. 它是怎么被制作出来的？【解决了用料问题，就要考虑建筑构成问题，这个是美术的基本素养，应该不需要解释太多】
3. 环境会对它产生什么影响？【这个就属于进一步的考虑了，一般专业的美术师傅在前期设计阶段就会思考环境（包括天气，光照，HDR图的选用等）会对建筑表现产生哪些影响，进而对结构和材质有什么样的要求。这些都是系统性思维，有经验的美术能够明确的根据设计要求拆分并量化出这些点给到其他环节的同事。经验弱一些的做起事情就会欠缺章法，采取走一步试一步的策略，而如果相当一部分工种都这样搞就会造成极大的成本浪费】
![](https://picx.zhimg.com/v2-5c9ae195d8f701ed24ae95700a78e691_1440w.jpg)

这些内容确定好，就可以准备纹理库和模型库了。对于CitySample，纹理部分使用了 [Megascans纹理库](https://zhida.zhihu.com/search?content_id=250765938&content_type=Article&match_order=1&q=Megascans%E7%BA%B9%E7%90%86%E5%BA%93&zhida_source=entity) ，但对一些特征进行了更改，大部分纹理为4k分辨率，并启用了“虚拟纹理”来解决内存中加载纹理数量限制的问题，对于一些更高分辨率要求的纹理使用了UDIM（可以简单理解为一种自动的UV偏移，将纹素分配到对应的UV图块中并建立起数字映射，虚幻引擎支持UDIM并且自动识别为虚拟纹理）。“ **每一种纹理都有明确的压缩类型** ，纹理流送池大小也被考虑进来以管理纹理加载的性能。这些纹理在Shader中采样时都考虑了“世界对齐”，并采用了真实的比例以保持足够的细节。 **每一类纹理都必须经过“PBR检测”，以验证“反照率，金属度，高光”等属性是否在合理的区间内** 。可能并不是所有材料都需要走“金属度流程”，因为相当多的材料完全不是金属质地，“高光流程”更适合一些，但无论采取哪种流程，只要过程严格遵守PBR生产准则，结果都是准确且安全的。如果没有单独开发”资产检查工具“，UE自带的 **Pixel Inspector** 也是不错的选择！纹理的生产最好有专业的”纹理设计师“负责，如果实在没有，在TA根据设计需求整理完相应的技术标准后，可由对应美术人员分别完成。

![](https://pic3.zhimg.com/v2-a4709b9c57b318a59231e85496963bde_1440w.jpg)

![](https://pica.zhimg.com/v2-e7d189b018d58c0c9b979b2017cc7e0a_1440w.jpg)

另外在UV上， **《黑客帝国》使用了两套UV** ，注意第二套UV不是烘焙用的，因为 [Lumen](https://zhida.zhihu.com/search?content_id=250765938&content_type=Article&match_order=1&q=Lumen&zhida_source=entity) 提供了全动态照明，纯PC上并不需要烘焙。

为了考虑和环境适应，材质上需要开发对应的功能并将功能封装为单独的材质函数以便实现逻辑的复用（比如幕墙水渍和一些建筑表面的污垢），这需要额外的环境Mask，这部分Megascans库并不包含，因此要单独烘焙：

![](https://picx.zhimg.com/v2-ea6d5885bdf27d77b9f4c334061fbcab_1440w.jpg)

各位可以看到，一切都有明确的规则与检验审计标准，一些设置是否启用都对应了明确的需求背景。

对于City场景来说，所谓的”基材“也就是”建筑材料“和”路网材料“，在考虑建筑材料时需要把窗户单独抽离出来，建筑主体做为另外一个独立的材料统一管理，那么就有了：

![](https://pic1.zhimg.com/v2-c667b47f7cf745f01c6ab80810dd0d34_1440w.jpg)

这些纹理,FBX数据将由” **AssetIngest Editor Utility** “控件自动批量的导入引擎，它会依靠PNG和FBX文件的刚性源文件结构，将纹理自动匹配到FBX对应的材质实例上。

### 1.2，建筑的模块化

在设计建筑样式时，前期一共考察了18种建筑：

![](https://pic4.zhimg.com/v2-2cb1478c9a4da58623f1bf4e58f7ca67_1440w.jpg)

**在设计层面有明确的建筑分类是非常重要的，它决定了后面能否进行合理的解构与抽象** 。因为建筑与城市都是程序化生成的，没有抽象的生成逻辑，程序化的意义便荡然无存。另外，我们不难发现，程序化生成与高度定制化的矛盾在这一环节就开始显露了出来，即： **如果希望通过PCG的方式生成场景，那么就必须摒弃“既要又要”的思维方式。且个人的操作习惯必须服务于整体的流程设计，而不是反过来为了少数人的“思维惯性”和“经验局限”去把管线和工具改的稀巴烂** 。

一旦建筑样式的范围被标定，我们便可以进一步从这些建筑样式中提炼出一些共性：比如这些建筑往往具有近似的窗户和外墙，因此非常适合实例化。基于这一点考虑，建筑被拆分为Modules（模组）以适配Nanite网格的性能要求和建筑多样性，只要分割合理便能组合出无数种独特的建筑造型。比如CHD中包含了72个Modules，CHC中包含了226个Modules，NYA中包含了485个Modules，这样一来就有超过2000个建筑模块。这些Module被进一步抽象为如下几类：

- 基础建筑模块（构成建筑）
- Corner 拐角：由几种不同的拐角组成
- Base Modules: 基础拐角
- Interior Corner 室内拐角
			- Exterior Corner 室外拐角
		- Split Corner：分割式左右拐角，方便制作锐角和钝角形状的拐角
- Corner L
			- Corner R
		- Caps Corner：角帽式拐角，类似于铰链结构，用于补偿分割式拐角在锐角和钝角时可能造成的穿模现象。大部分情况下为“直角角帽”
	- Wall 墙壁
	- Entrance 入口
- 额外模块（用于打造更复杂的建筑墙面）
- WallCap 墙帽：用于装饰重复的墙体模块，区分建筑左右
	- Transition 过渡模块：可以缩放，用于填补各类间隙
	- Pillar / Column 立柱：一般放置在窗口或者墙体之间，理论上不能缩放
![](https://picx.zhimg.com/v2-fab2e70f8c221f417b5c0072f3eb8019_1440w.jpg)

这里有一个隐性的前提：就是该模块划分是建立在一定细节精度的要求和Nanite管线的技术前提下的。通俗点说就是“我们希望建筑能够保留尽可能多的细节，同时因为我们所有资产都会走Nanite管线，所以可以不在乎面数”。但倘若去掉Nanite这一技术前提，面数，游戏视角和Culling策略便是必须权衡的三类因素。这也就意味着可能不得不选取更粗一些的模块化策略，从而将设计限定在一个更窄的发挥范围内。

确定所需的模块后，使用“ **Houdini Module Template Generator** ”来生成代理模块，然后在Maya中以这些生成的建模为起点制作高分辨率几何体。由于许多建筑包含数百个Module，为了保证团队效率开发了一个“ **自定义导入器** ”批量导入建筑模块，创建必要的文件夹结构，并为每个模块创建和指定材质实例。另一方面，为了保证美术能控制建筑墙面的程序化生成方式，创建了一套被称为“ **形状语法** ”的表达式语言，用于描述墙面的构造方法。形状语法支持显示的描述建筑模块，并支持模块重复，模块循环，以及定义哪些模块能够缩放和建筑垂直部分的详尽表达。比如下图2蓝色模块为建筑的最底层，需要显示指定位置。红色部分由于不能重复出现，所以也要显式的指定。绿色部分则可以重复。最后，紫色充当楼顶用于结束建筑的垂直搭建。 **垂直形状语法** 旨在定义哪些楼层可以重复使用，哪些部分可以分成不同风格以及如何放置顶部模块来完成墙体部分。这些形状语法会和层级语法记录在 **JSON** 格式的建筑定义文件内，该文件会从虚幻引擎外部保存，然后在Houdini中构建程序化建筑

![](https://pic1.zhimg.com/v2-39cc8d1a00bcf1fe02d81d7a284aee64_1440w.jpg)

![](https://picx.zhimg.com/v2-5e15f25dff0982207f9755a6d84c8579_1440w.jpg)

### 1.3，建筑道具挂件

为了让生成的建筑更具备一些美感， **在模块上手动添加了一些道具锚点用于挂载装饰物，道具会被注册到模块并分组，在城市的程序化生成过程中随机选取** 。道具按类型分为：

- 雨篷
- 灯具
- 路牌
- 招牌

这样一来，记录模块元数据就非常重要， **元数据由Houdini Template Generator生成并存储，Houdini Engine会将这些模块的Attributes转换为标签（Tags）并存储到静态网格对象上以后管线的后续流程中轻松获取** 。另外，建筑模块竖轴点位置也很重要，因为涉及到建筑的对齐，在黑客帝国案例中枢轴点被设定在Module的左侧和正面外墙的边线对齐。建筑的风格被记录在 **BDF** 内，它代表了构建某种建筑风格需要的所有东西。

建筑模块信息和风格信息会被打包送入Houdini进入“程序化建筑生成环节”。“ **建筑生成器** ”这个HDA会同时在Houdini和UE5中使用。 **它还会为建筑程序化生成一套边缘网格，以便管线后期进行渲染时能更好的与地面融合在一起** ！

![](https://pic4.zhimg.com/v2-ac820fd13db7de953a3fff36073e3157_1440w.jpg)

### 1.4，建筑窗户风格

1.2和1.3讨论了建筑的模块化和建筑道具挂点，建筑另外一个很重要的组成是窗户，透过窗户内景我们能进一步区分出建筑的功能分类，同时增加建筑表现的丰富度。窗户由单独的材质表现，涉及到 **Interior Mapping** 机制，关于材质设计和渲染原理这里暂且不表。房间的数据会被打包成32位的无符号整型传送给Houdini和UE插件以实现分类，所有材质需要的信息上都有标志方便什么类型的房间需要渲染窗户，这些数据为每个实例的自定义数据（也就是建筑ISM中的每个实例），然后在材质中使用这些数据。

![](https://pic4.zhimg.com/v2-3e397346ce8fe8bcf2e0f8346f297dc1_1440w.jpg)

### 1.5，主建筑和群落概念

还创建了一系列工具用来在城市街道和建筑屋顶上散布细节，这些内容被统称为“ **人工资产群落** ”（官方叫法“ **Biomes** ”）。这些“群落工具”使用了UE5的“ **Level Instance Packed Blueprintes** ”(关卡实例打包蓝图)，打包蓝图能基于任意选定的Actor创建，选中的Actor会被导出到新创建的关卡且其中只包含被选中的资产，所有的Actor都会转成 **ISM组件** 包含在打包蓝图中。这种方法的一个好处是可以将打包蓝图放置在关卡中，然后再进行编辑。通过这种方法创建出了城市中的所有群落，主建筑和主区域：

![](https://picx.zhimg.com/v2-1313b9028e7793fd180d7d78d3c14321_1440w.jpg)

这些主建筑和群落也会被送入 **City Generator（城市生成器）** 中用于城市中的程序化内容生成。

## 二，引擎内资产管理

### 2.1，资产管理

城市生成之后，点云信息就会被保存到 **Alembic** 文件中，这些 [Alembic文件](https://zhida.zhihu.com/search?content_id=250765938&content_type=Article&match_order=1&q=Alembic%E6%96%87%E4%BB%B6&zhida_source=entity) 被称为 **PBC** ，因为它只包含了点信息以及创建建筑，街道和碰撞物所需的程序化几何体，并不包含完整的城市。这个PBC会被导回虚幻引擎，然后使用 **Rule Processor** 生成城市。

整个城市的构成非常复杂，包含了：

- 建筑
- 街道
- 贴花
- 人群和车辆
- 声音

等海量信息，为了管理这些Actor，UE5引擎这边配套了“ **[World Partition](https://zhida.zhihu.com/search?content_id=250765938&content_type=Article&match_order=1&q=World+Partition&zhida_source=entity)** ”和“ **OFPA** ”机制，前者用于在编辑器和运行时动态加载对象，后者将加载与卸载粒度精细化到Actor层级，将Actor和关卡解耦以支持同一关卡的多人同时编辑。与World Partition配套的资产管理方案还有“ **[Data Layer](https://zhida.zhihu.com/search?content_id=250765938&content_type=Article&match_order=1&q=Data+Layer&zhida_source=entity)** ”, 将Actor按标签分层管理，一个Actor可以同时身处多层，也可以仅存在单独的某一个数据层。Data Layer又分为编辑器数据层和运行时数据层，前者仅用于管理编辑器场景并且没有运行时开销。

### 2.2，HLOD策略

整座城市包含了一个“Main Grid”和两个“HLOD Grid”，Main Grid 包含了距离玩家128米内的所有资产；HLOD0则为128~768米，每个Grid Cell包含了4个HLOD0的Actor，即Main Grid中的所有Actor都会合并到四个独立的HLOD0 Actor内，并整合到ISM组件，这样就可以减少Actor的总数并改善单元流送，又因为HLOD0 Actor是由Main Grid中的Actor生成的，因此当Main Grid中的Actor做出修改时HLOD0都需要重新生成，这个频率在团队开发时被设定为通过“自动化流程”每晚生成一次；超过768米范围就是HLOD1了，它始终会在内存中加载以构成城市天际线和远景，HLOD1会将一个单元内的所有静态网格体都合并成一个Nanite网格，与HLOD0类似前面层次的任何更改都会触发HLOD1层级的重新生成。无论是程序化生成的建筑还是手动制作的建筑都使用ISM组件来搭建，这样能在设计过程中尽可能减少城市中Actor总数并帮助改善流送性能。

### 2.3，碰撞体策略

在碰撞处理上，建筑的上部分和下部分也区分了策略，下部分使用精确到模块的碰撞体以提供精准的碰撞效果，这样玩家才能靠近内陷的部分，走过有天顶的过道和走廊。而对于建筑的上半部分，则仅用了碰撞，使用了一个独立的基本碰撞对象来代表整篇区域，并且作为单独的Actor放置，这样可以做到减少碰撞计算。

### 2.4，光追分组

为了优化光线追踪下的表现，还给每个对象都建立了光线追踪分组，以便优化Lumen的Surface Cache。在设置分组时，原则上会避免包含稀疏分布的对象，因为光线追踪分组的目的就是为了让靠近的对象组合在一起减少光线求交的频率。网格体也要尽可能减少重叠，因为如果重叠过多也会拖慢硬件光追的求交速度。

### 2.5，Rule Processor

**Rule Processor（规则处理器）** 是一套插件【 **注意不是引擎内部插件，而是随CitySample Release的项目插件** 】，它可以通过Alembic导入点云数据。点云可以来自任何DCC工具，在《黑客帝国：觉醒》项目中这个DCC为Houdini。规则可以很简单，比如每个点都生成一个Actor；规则也可以很复杂。整个城市有多个规则组织而成，显然构成建筑和构成道路的Actor就得遵循两套不一样的规则。Rule Processor能追踪Actor，会清理掉就Rule下的Actor（智能回收）；Rule Processor还能查看元数据关键值，并将数据作为Actor的属性覆盖，当你要为Actor设置某些特殊属性比如对网格体贴花和碰撞对象，这个功能就很好用。
