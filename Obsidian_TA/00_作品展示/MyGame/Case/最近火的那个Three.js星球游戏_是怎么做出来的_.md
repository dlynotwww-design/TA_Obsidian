# 最近火的那个Three.js星球游戏，是怎么做出来的？

作者: 毛豆西葫芦园子

发布时间: 2026年6月19日 16:51

发布地点: 上海

# 背景

最近社交平台上有一个 Three.js 做的 3D 星球页面火了——打开网页就是一个低面数（low-poly）风格的外星世界，有行星、水面、云层、树木、瀑布，你可以旋转视角到处探索。画面干净、氛围感很强，很多人转发问「这是怎么做的」。

这个作品叫 Messenger，作者是独立开发者 abeto。它不是一个游戏——没有任务、没有战斗、没有关卡——而是一段「可交互的视觉叙事」，纯靠 3D 渲染和氛围讲故事。

很多人看到这个效果的第一反应是「这得一个团队做半年吧」。其实拆开看，核心就几块，一个人完全搞得定。

messenger-copy 是一个开源的学习项目，用 SvelteKit + Three.js 从零复现了 Messenger 的视觉效果，代码完整可跑。仓库的 ATTRIBUTION.md 声明：原作品美术、音频、设计版权归 abeto 所有；仓库代码（src/）MIT 开源，大部分是 AI 辅助生成的——一个人用 AI 工具写出来的。

这篇文章就对照 messenger-copy 源码，讲清楚这类 WebGL 视觉效果是怎么做出来的。看完你应该能自己动手做一个。

# 先看一眼技术栈

层面 | 技术
--- | ---
前端框架 | SvelteKit 2 / Svelte 5
3D 渲染 | Three.js r184
语言 | TypeScript
样式 | Tailwind CSS v4
测试 | Vitest + Playwright
包管理 | pnpm

代码量分布藏着关键信息：GLSL 着色器代码占 79%，TypeScript 14%，Svelte 组件只有 3.5%。这说明一件事——视觉效果的核心不在框架，在着色器（shader，跑在 GPU 上的渲染程序）。框架只是壳，着色器才是灵魂。

# 第一步：搭一个 Three.js 场景壳子

不管效果多炫，第一步永远是让画面先出来——一个能转视角的 3D 场景。

messenger-copy 的场景入口是 MessengerScene.svelte，做的事很标准：

创建 Scene → 放摄像机 → 加灯光 → 挂 OrbitControls → 启动渲染循环

几个要点：

- 摄像机用 PerspectiveCamera，视场角 42°，不用太广角，画面会更稳
- 灯光两盏起步：一盏 AmbientLight（环境光，全局提亮，不让暗面死黑），一盏 DirectionalLight（方向光，暖白 #fff1dc，制造体积感和阴影方向）
- OrbitControls（轨道控制器）让用户拖拽旋转、滚轮缩放。设了阻尼（damping），让旋转有惯性手感；加距离限制，不让镜头跑出画面
- 渲染循环用 requestAnimationFrame，每帧更新控制器、更新材质 uniform（GPU 端变量）、然后调用渲染器绘制

这一步做完，你看到的是一个空荡荡的灰色背景。别急，画面是逐层叠加出来的。

# 第二步：加载 3D 模型

场景需要物体——行星、树木、水面、云层、瀑布。这些 3D 模型从哪里来？

messenger-copy 覆盖了三种场景模式，每种加载不同的模型组合：

场景 | 加载的模型
--- | ---
Intro | 行星、水面、树木、云层（4 个 mesh）
Gameplay | 地形、树木、水面、电缆、瀑布本体、瀑布溅射、瀑布入水口、烟雾、海滩泡沫（9 个 mesh）
NPC | 带骨骼动画的角色模型

每个模型文件是一个压缩过的 3D 几何体（geometry，就是顶点位置 + 面片索引 + 法线方向的数据）。messenger-copy 用了两种加载器：

- Draco 压缩几何体（.drc） — Google 开发的 3D 压缩算法，压缩比通常 10:1 以上。一个几十 MB 的模型压到几 MB，对 Web 加载至关重要。Three.js 的 DRACOLoader 在后台线程（Web Worker）里解压，不卡主线程。
- KTX2 纹理（.ktx2） — Khronos Group 的 GPU 压缩纹理格式。好处是纹理在显存中保持压缩状态，省显存；直接支持 BCn/ASTC 硬件解码，跳过 CPU 解压。

模型加载完后，需要把每个 geometry 包装成 Mesh 挂到场景 Group 上——geometry 提供形状，material 提供外观（颜色、纹理、光照反应）。接下来讲材质才是重点。

# 第三步：写着色器——画面的灵魂

几何体只是形状，着色器决定它长什么样。messenger-copy 96% 的代码是 GLSL 着色器，剩下的 TypeScript 是胶水代码。这就是为什么这类效果看着复杂——工作量在着色器上。

着色器分两种：

- 顶点着色器（vertex shader）：处理每个顶点的位置，可以位移顶点做动画（比如风吹树叶）
- 片元着色器（fragment shader）：处理每个像素的颜色，可以采样纹理、计算光照、混合颜色

messenger-copy 有 5 个核心着色器，照搬过来你就能复现效果：

**水面着色器（gameplayWater.glsl）**

最出效果也最复杂的一个。拆开来看其实就是几层叠加：

- 波纹动画：用时间 uniform 驱动多层 simplex noise（一种平滑的伪随机函数），让水面顶点随时间上下浮动
- 深度透明：从场景深度图里读取每个像素到摄像机的距离，水深的地方不透明、水浅的地方透明——这就是水面边缘有「浅滩感」的原因
- 菲涅尔效应（Fresnel）：视线和法线夹角越大，反射越强。翻译成人话：正对着看水面是透的，斜着看水面反光

**地形材质（terrainMaterial.glsl）**

同样靠噪声。不同之处在于：

- 多层噪声叠加：用不同尺度（octave）的噪声混合，大尺度的噪声做大的地形起伏，小尺度的做表面细节
- 按高度着色：噪声值高的地方是高地（深色），低的地方是洼地（浅色），自然形成地形层次感
- 纹理图集（atlas）映射：地面纹理不单独加载多张图，而是拼在一张大图里，着色器按坐标取对应区域

**树叶动画（treeLeaves.glsl）**

这个是纯顶点着色器的活——不用改颜色，只改位置：

- 每个顶点沿法线方向（顶点在模型表面往外指的方向）上下位移
- 位移量用噪声和时间驱动，不同顶点位移幅度不同
- 结果就是树叶在「呼吸」——整片树叶不规则地微微晃动，像被风吹

**行星 / 云层材质（messengerMaterials.glsl）**

云层的技巧是噪声驱动 alpha：用 simplex noise 决定每个像素的透明度。噪声值高的地方不透明（云的白肚子），噪声值低的地方透明（云的边缘），过渡自然不生硬。

行星材质是标准的纹理采样 + 漫反射光照——贴一张行星表面纹理图，按光照方向算明暗。

**后处理着色器（postProcessing.glsl）**

场景渲染完后不是直接输出屏幕，而是再套一层滤镜：

- 把渲染结果作为输入纹理
- 结合场景深度图
- 用 LUT（色彩查找表，其实就是一张颜色映射图）做全局色彩分级——相当于一键套 Instagram 滤镜

整个渲染管线是：

3D场景 → RenderPass（渲染到离屏缓冲区） → ShaderPass（LUT色彩分级+深度效果） → 屏幕

# 第四步：用噪声驱动一切

回顾上面所有着色器，会发现一个共同的模式：噪声函数无处不在。

Simplex noise 是一个生成平滑伪随机值的数学函数，输入一个坐标，输出一个 0~1 之间的值——相邻坐标的值接近，离远了的坐标值随机。就像连绵起伏的山丘，不是纯随机的白噪声。

在这个项目里，噪声扮演的角色：

用途 | 怎么用
--- | ---
水面波纹 | 噪声驱动顶点位移，时间参数让噪声「流动」起来
云层透明度 | 噪声输出直接当 alpha 值
地形起伏 | 多层噪声（不同频率振幅）叠加，生成自然地面
树叶摆动 | 噪声驱动每个顶点的位移幅度
道具颜色变化 | 噪声给均匀颜色加一点不规则变化，避免塑料感

这是独立作品里的经典做法——用数学函数替代大量美术资源。一个熟练的开发者只要几 KB 的着色器代码，就能生成一个看起来充满了手工细节的场景。

# 第五步：组装和性能优化

所有零件都有了，最后一步是组装并让它跑得流畅。

**场景拆分与懒加载。** 不用一次性加载所有模型，而是按场景模式动态 import。messenger-copy 用了一个令牌（token）机制：发起加载时记录一个 token，加载完成时检查 token 是否还是最新的——如果用户快速切换场景，旧请求的结果直接丢弃，画面不会闪烁。

**自动适配尺寸。** fitGroup 函数遍历所有 mesh，算出合并包围盒，然后等比缩放到目标尺寸——这样不管原始模型多大，场景始终占满画面。

**后处理性能。** 后处理本质是多渲染一遍，会吃掉一些帧率。如果目标设备是手机，可以跳过 LUT 通道直接输出，画面差一点但能跑满 60fps。messenger-copy 把后处理做成了可选链路，开关很方便。

**浏览器测试。** 注意测试依赖里有 @vitest/browser-playwright——测试在真实浏览器里跑。因为 Three.js 渲染依赖 WebGL 上下文（GPU 接口），Node 环境没有 GPU，只有浏览器里才能验证渲染管线。

# 总结

这类 WebGL 视觉效果，拆开来看就是五步：

1. 搭 Three.js 场景壳子（摄像机、灯光、控制器、渲染循环）
2. 加载 3D 模型（Draco 压缩几何体 + KTX2 纹理）
3. 写着色器（水面、地形、树叶、云层、后处理——噪声函数是核心）
4. 噪声驱动一切（用数学替代美术资源）
5. 组装优化（懒加载、动态缩放、可选后处理、浏览器测试）

messenger-copy 证明了一件事：一个人 + AI 辅助，完全可以从零做出这种效果的网页。AI 帮忙写了胶水代码和着色器框架，人负责理解原理和做审美决策。

如果你也想做一个，建议从最简单的开始——先搭一个只有一个球 + 一盏灯的场景，然后逐步加着色器、加后处理、加模型。不要一上来就想做完美，每加一层多一个效果，慢慢就会变成你看到的那个样子。
