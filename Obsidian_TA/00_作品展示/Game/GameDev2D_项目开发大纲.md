### 项目基本定位

- **游戏类型**：2D像素风、休闲收集（点击/拖动）
- **核心体验**：治愈、宁静、收集欲
- **目标平台**：PC（Steam） / 移动端（TapTap）
- **分辨率基准**：以 **16:9** 为画幅，像素单位采用 **16x16** 或 **32x32** 为基础网格。

## 策划

### 玩法

策划在这个项目中，核心是**做减法**，确保"零压力"。

### 1. 核心循环机制
- **操作层**：鼠标/手指拖拽玻璃罐移动，罐子自带圆形吸附光环。
- **反馈层**：萤火虫进入光环后，伴有"叮"的音效和像素粒子爆开，数量+1。
- **成长层**：收集满指定数量（如 50只），解锁"森林新区域"或"新萤火虫品种"。

### 2. 数值与难度曲线（宽松向）
- **刷新密度**：根据罐子停留时间动态调整——停在原地超过3秒，周围萤火虫刷出率提升20%（鼓励挂机发呆）。
- **收集阈值**：前3种普通虫阈值 10/20/30 只；稀有虫（随机乱跑）只需捕捉 5只即可解锁图鉴。
- **经济系统**（若有）：收集的萤火虫可转化为"星光币"，用来购买不同样式的罐子（纯外观，无属性加成）。

### 3. 关卡/区域设计
- **森林**（初始）：地面平坦，萤火虫移动缓慢。
- **溪流**（第二关）：有动态水流像素块，萤火虫会顺水飘动。
- **花丛**（隐藏关）：需要集齐前两关图鉴解锁，萤火虫呈"8"字飞行轨迹。

### 4. 易用性与引导
- 无需新手教程，进入游戏只用一行像素文字提示："拖动罐子，收集光"。
- 增加"禅意模式"：关闭UI数字显示，只有画面和声音。

### 架构设计

程序需要保证游戏轻量、流畅，且像素画面"不糊"。

### 1. 引擎与渲染设置
- **推荐引擎**：Unity（2D URP）或 Godot（2D专用）。
- **像素完美（Pixel Perfect）**：**必须**挂载 `Pixel Perfect Camera` 组件，将缩放倍数设为整数（如 x3 或 x4），防止像素点在屏幕上产生扭曲拉伸。
- **分辨率适配**：以 960x540（2倍）或 1920x1080 为基准，锁定缩放比例，上下或左右增加黑边/装饰边框。

### 2. 核心逻辑模块
- **对象池（Object Pool）**：萤火虫和收集粒子频繁生成销毁，**禁止**使用 `Instantiate/Destroy`，必须使用对象池循环复用，避免GC卡顿。
- **AI移动算法**：
  - 普通虫：`Perlin Noise（柏林噪声）` 漂移，速度慢，随机转向。
  - 稀有虫：加入 `SmoothDamp` 平滑追逐罐子，但到达一定距离（安全半径）立即弹开，制造"捉迷藏"乐趣。
- **吸附判定**：以罐子中心为圆心做 **OverlapCircle** 检测，进入半径的虫子向罐子插值移动（Lerp），无须复杂的物理刚体，直接修改Transform即可。

### 3. 数据持久化（存档）
- 使用 **JsonUtility** 将玩家数据（收集数量、解锁图鉴、拥有的罐子皮肤）序列化为JSON，存储在 `Application.persistentDataPath`。
- 存档时机：**每收集到 5只 自动保存一次**，防止玩家忘记存档。

### 4. 性能优化（针对移动端）
- 限制最大粒子数量（如 200个），超出则回收最早的粒子。
- 背景视差图块采用 **Static Batch（静态合批）**，减少DrawCall。
- 关闭不必要的物理引擎（取消重力，设为2D Space）。

## 美术

### 美术参考
### 免费资产网站
### AI资产平台

像素风的核心在于**限制**，在限制中做出识别度。

### 1. 像素规格与色板
- **网格限制**：所有角色和主要物体必须基于 **16x16** 或 **32x32** 网格绘制（推荐角色用32x32，环境图块用16x16）。
- **色板限制**：定义一个**全局调色板（Palette）**，控制在 **24-32色** 以内（避免颜色太杂失去像素味）。例如：
  - 夜晚背景：深蓝（#0A0E27）、藏青（#1B2A4A）
  - 萤火虫：明黄（#FFE66D）、暖绿（#A8E06C）
  - 罐子：高光白、玻璃淡蓝（半透明叠加效果靠程序处理，美术只画基础质感）。

### 2. 具体资产列表
- **环境（Tilemap）**：
  - 地面草地（要有高低起伏变化，至少画 4种不同草地块用于随机拼接）。
  - 背景树木、灌木丛（做 **Parallax（视差）** 分层：近景暗、中景稍亮）。
- **角色/物体（Sprite）**：
  - 玻璃罐（带光影高光像素点）。
  - 至少 6种颜色的萤火虫（每种颜色要有 **2帧待机闪烁动画** 和 **1帧被捕瞬间的"！"表情**）。
- **UI（像素字体）**：使用商用免费的像素字体（如 `Pixelify Sans`），或美术手绘 0-9 数字用于显示数量。
- **特效（粒子）**：萤火虫被收集时爆开的 **4向星芒粒子**（只需画 4张单帧像素图，由程序控制旋转缩放）。

### 3. 风格统一要点
- 采用 **"暗调剪影+发光核心"** 的风格。背景树木尽量用剪影（纯黑或极深蓝），只有萤火虫和罐子是鲜艳的，这样玩家视觉焦点不会被抢走。

## 里程碑建议（分步走）

### 灰盒阶段

程序用方块代替萤火虫实现移动和吸附，策划验证手感。美术只出 **1张概念氛围图** 确定色调。

在游戏开发中，**"灰盒阶段"（Graybox / Prototype Phase）**的核心目标是**"验证核心玩法好不好玩"**，而不是"画面好不好看"。因此，这个阶段所有岗位的产出都必须是**"快速、可迭代、容错率高"**的。

#### 📝 策划产出：可调参数表 + 核心循环验证报告

策划在这个阶段不是写长篇大论的文档，而是**出一份"体验调参表"**，并每天进行玩法验证。

- **核心循环说明书（一页纸）**：用最简单的流程图描述"拖拽罐子 → 虫子进入范围 → 数量+1"的闭环，确保程序和美术理解一致。
- **可调参数表（Excel 或 引擎内配置）**：将影响"手感"的关键参数全部列出来，并给出初始建议值，方便在引擎中直接调整。
  - *吸附半径*：建议初始 2.5 个逻辑单位（太大没挑战，太小捉不到）。
  - *虫子移动速度*：建议初始 1.0 单位/秒。
  - *虫子刷新密度*：建议初始 每5秒/只。
  - *收集阈值*：第一关解锁条件设为 10只。
- **灰盒地图布局草图**：不需要画精美的像素画，只需用**方块**在纸上或PPT里标出"森林关"的边界、玩家出生点、树木阻挡区域（如果有的话）的位置。
- **每日体验报告**：每天下班前玩半个小时当前的灰盒版本，记录"手感是否太滑"、"虫子是否太密"，并调整参数表发给程序。

#### 💻 程序产出：项目结构 + 可运行的"丑陋"原型 + 调试面板

程序在灰盒阶段的目标只有一个：**让游戏能跑起来，并且策划能随时改参数**。

- **可执行灰盒Demo（核心产出）**：一个可以在PC或手机上运行的安装包。所有物体都是灰块或几何体，但**核心逻辑必须完整打通**：
  - 鼠标/手指拖拽罐子（灰块）移动。
  - 虫子（灰点）在场景中做柏林噪声漂移。
  - 当灰点进入灰块的范围时，灰点消失，屏幕左上角的数字 +1。
- **对象池系统（底层架构）**：虽然场景是灰盒，但程序必须把"对象池（Object Pool）"写好，保证虫子销毁时不会产生内存垃圾，避免后期卡顿。
- **实时调试面板（重中之重）**：在游戏屏幕上或Inspector面板中，把策划要调的参数**全部暴露**出来。程序写完代码后，策划不需要重启游戏，就能在运行中滑动滑块改变：
  - `CatchRadius`（吸附半径）
  - `SpawnInterval`（生成间隔）
  - `BugSpeed`（虫子速度）
- **基础的存档占位**：写一个最简单的 `PlayerPrefs` 或 JSON 存档逻辑，哪怕只存一个"当前收集总数"，确保数据流转没问题。

#### 🎨 美术产出：技术规格规范 + 基础灰盒资源包

美术在灰盒阶段**不用画最终的精美像素画**，但需要做两件极其重要的事：**定"尺子"** 和 **做"临时替身"**。

- **像素规格白皮书（关键产出）**：一份写给程序和自己的技术文档，确定所有硬性指标。
  - *世界单位与像素比*：规定 **1个逻辑单位 = 16像素**（或32像素），这样程序在做碰撞时，美术知道画多大。
  - *锚点规则*：规定所有物体（罐子、虫子）的Sprite锚点必须统一在 **底部中心（Bottom Center）**，方便程序做旋转。
- **概念氛围图（1张）**：画一张**静态的、高完成度的概念插画**（即使灰盒里实现不了光照，这张图也是全组灰盒阶段的"灯塔"，决定最终的色调是冷是暖）。
- **灰盒资源包（替身素材）**：不需要画萤火虫，而是快速产出：
  - 一个 **32x32 的灰色圆点**（代表萤火虫）。
  - 一个 **64x64 的灰色方块带半透明圈**（代表玻璃罐吸附范围）。
  - 几种不同深浅灰色的 **16x16 方块**（代表树木、草丛、地面的阻挡块）。
  - *注意*：不同物体最好用不同的明度灰色（如罐子用深灰，虫子用浅灰），方便在引擎中肉眼区分。

#### 总结一句话：
灰盒阶段，**策划产出"一堆数字"**（参数表），**美术产出"一张画"**（概念图）和**"一堆灰色方块"**（占位图），**程序产出"一个丑游戏"**（可运行的Demo）。这三样东西合在一起，如果玩起来**"意外地有点上头"**，那灰盒阶段就完美收官了！
